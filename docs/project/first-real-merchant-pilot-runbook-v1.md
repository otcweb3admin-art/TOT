# First Real Merchant Pilot Runbook V1（首家真实商家试运行·准备与执行手册）

> 文档类型：纯业务 / 产品运营 / 系统操作 Runbook（TASK-076）。不改代码、不建真实商家、不建真实账号、不写真实客户信息。
> 上游依据：UAT 全链彩排报告（PASS）、账户分权工作流设计 V1、多 AI 蓝图 V1、Readiness Gate（Conditional Go）、Outreach Kit、Prospect Tracker、Field Pack、内部账号指南、角色交接指南、证据宪法 E0–E3、生产 RC 验收。
> 最高约束：**AI 不拍板（Human Commercial Authority）**——是否接入某家真实商家、何时接入、收费与资源投入，均由人工商业负责人决定；本手册只规定"决定之后怎么按固定顺序执行"。

---

## Chapter 1：Executive Summary（执行摘要）

系统已通过 **UAT 全链彩排（PASS，2026-06-12）**：采集 → 人工审核 → AI 草稿审核 → 人工保存节点 → 外包提交/验收 → 客户确认/反馈 → 审核员收口，可作为一条完整流程运转。但**真实商家接入仍是商业决策，不是自动动作**——系统就绪 ≠ 自动开始。

本 Runbook 的目的：首家真实商家进入系统时，**不再临时摸索，而是按固定顺序执行**——谁建账号、谁建商家、谁采集、谁审核、谁调 AI、谁分外包、谁验收、谁让客户确认、哪些数据能录、哪些必须标待验证、哪些动作禁止自动化、首单怎么复盘，全部有章可循、可交接给任何接手的人。

**核心原则（九条，全文反复出现，违反即停）**：
1. 先授权，再录入。
2. 先采集，再诊断。
3. 先审核，再进入下一步。
4. AI 只出草稿，人工审核（AI 永不拍板、永不直写业务节点）。
5. 外包只看必要任务，不看完整经营数据。
6. 客户只看进度、待配合、待确认事项（不看内部 Prompt / 外包讨论 / 审核争议）。
7. 不承诺增长结果（不承诺 ROI / 客流 / 转化 / 销售额）。
8. 不把口述当事实（口述标口述、估算标估算、不确定标待验证——证据宪法 E0–E3）。
9. 不把 UAT / DEMO 当真实案例。

## Chapter 2：Pilot Entry Criteria（进入条件）

首家真实商家进入系统前，**必须全部满足**以下条件（对照 Outreach Kit 第 8 章 + Readiness Gate 放行条件）：

| # | 条件 | 确认方式 |
|---|------|---------|
| 1 | 商家**明确同意**参与试运行 | 负责人与商家当面/书面确认，记录日期 |
| 2 | 商家知道这是**线上代运营增长服务试运行，不承诺结果** | 按 Outreach Kit 承诺边界话术讲清并确认 |
| 3 | 商家愿意提供基础资料（按 Field Pack 清单） | 面谈确认可提供项 |
| 4 | 商家愿意配合确认内容（客户确认事项） | 说明"确认通过我们才进下一步"并获认可 |
| 5 | 商家能接受 **人工 + AI + 外包协同** 模式 | 明确告知 AI 出草稿人工审核、部分制作外包 |
| 6 | 商家同意**资料仅用于本项目服务** | 口头/书面确认，不另作他用 |
| 7 | 内部已指定**项目负责人**（人） | 写入跟进表（Prospect Tracker） |
| 8 | 内部已备好 采集员 / 审核员 / 外包 / 客户 账号角色 | 按 Chapter 3 检查 |
| 9 | **不录入未授权的商家真实隐私信息** | 授权范围之外的一律不进系统 |
| 10 | **沟通意向 ≠ 正式授权** | 只有明确授权才算满足条件 1 |

**结论规则：任一条不满足 → 不进入系统正式建真实商家。** 可继续用 Outreach Kit / Prospect Tracker 推进沟通，或用 UAT_/DEMO_ 数据演示，但不得建真实 Merchant。

## Chapter 3：Account Preparation（账号准备）

按内部账号指南（internal-role-account-setup-guide-v1）人工在 Supabase 创建并设置角色；**密码线下交付，绝不写入仓库/聊天/文档**。每类账号一人一号。

| 账号 | role | 用途 | 红线 |
|------|------|------|------|
| **平台管理** | `admin` | 账号与角色管理、全局查看、异常处理 | **不作为日常运营账号；不多人共享**；不用于日常采集/审核/外包执行 |
| **人工审核** | `operator` | 核心运营审核：审采集、用 AI 工作台、审 AI 草稿、建/分/验外包任务、发起与处理客户确认、推进任务 | 不绕过证据纪律；received ≠ approved；没证据不通过 |
| **采集员** | `collector` | 新建商家、采集原始信息、填 Profile / Baseline / Operating Capacity、提交采集任务 | 不做最终诊断、不承诺结果、不编数据（不知道写待补充） |
| **外包** | `executor` | 看分配给自己的外包任务、看要求与验收标准、提交成果链接/说明、按退回意见修改 | 只见自己的任务；不见完整经营数据；不直接联系客户（除非授权） |
| **客户** | `merchant` | 看自己的事项、看待确认内容、确认通过、提出修改意见 | 不见内部任务/AI 草稿/外包信息；不可创建内部任务 |
| **AI Worker** | `ai_worker` | **不作为真人账号**——AI 能力由审核员通过「AI 工作台」调用 | 不开账号给真人；登录会看到说明页 |

注：当前 `admin@tot.local` 实际 role=operator（临时夹具）；正式试运行前按账号指南换为真实账号体系并改密/停用夹具（人工决策与操作）。

## Chapter 4：First Merchant Creation Rules（真实商家创建规则）

- **谁可创建**：仅 `admin / collector / operator`（系统权限即如此；executor / merchant / ai_worker 不可）。
- **何时创建**：Chapter 2 十条全满足、**商家授权之后**。授权前不得创建真实商家。
- **命名**：必须用真实商家名称，**不加 UAT_ / DEMO_ / SMOKE_TEST_ 前缀**；反之，凡演示/测试**必须**用 UAT_ / DEMO_ 前缀——前缀是系统识别"非真实数据"的唯一标记，混用即污染。
- **录入纪律**（Field Pack + 证据宪法）：
  - 真实资料**标明来源**（谁说的 / 哪份单据 / 哪个后台截图）；
  - 口述信息标「口述」；估算数据标「估算」；
  - 不确定信息一律写「待补充 / 待验证」——**宁缺勿编**；
  - 客户隐私只录授权范围内、服务所需的最小集合。

## Chapter 5：Pilot Operating Sequence（首单完整执行顺序）

按序执行，**每步完成才进下一步**；括号内为系统位置/任务类型。

1. **商家授权确认**（负责人，线下；记录到跟进表）
2. **建立客户账户**（admin 按账号指南建 merchant 账号）
3. **建立内部角色账号**（如尚未存在：collector / operator / executor）
4. **创建真实商家 Merchant**（collector 或 operator：`/dashboard/merchants/new`）
5. **创建采集任务** `collector_intake`（任务中心「新建任务」，负责人角色=collector）
6. **采集员填写基础资料**（先 Field Pack 线下采集，再按接入向导录 Profile / Baseline / Operating Capacity）
7. **采集员提交采集任务**（任务详情「开始任务」→「提交审核」，成果摘要写清采集了什么）
8. **审核员审核采集资料**（任务中心待审核队列；对照证据来源）
9. **不完整则退回**（「退回修改」+ 必填修改意见 → 采集员补充重交）
10. **通过后进入 AI 诊断草稿**（审核员「审核通过」→「标记完成」）
11. **AI 工作台生成 Prompt**（operator：`/dashboard/ai-workbench`，选商家 + 商家诊断草稿）
12. **人工用外部 AI 生成草稿**（复制 Prompt 到自己使用的 AI 工具）
13. **粘贴回系统**（回填区粘贴 + 人工修改：删编造、标待验证）
14. **提交为 AI 草稿审核任务**（按钮一键创建 `ai_draft_review`，状态=待审核）
15. **审核员审核 AI 草稿**（对照验收标准：事实有据、缺口标注、无承诺）
16. **通过后人工保存到对应业务节点**（详情页「去 TB-001 编辑页手动保存」——**系统不自动写**）
17. **创建外包任务**（如需图片/视频/文案：`outsource_execution`，写清要求+验收标准，分配 executor）
18. **外包提交成果**（成果说明*/链接/备注 →「提交成果，等待审核」）
19. **审核员验收成果**（通过 / 退回带意见 → 外包重交）
20. **通过后创建客户确认事项**（`client_confirmation`，给客户看的语言；分配客户账号 → 开始 → 提交=发起确认）
21. **客户确认或提修改意见**（客户登录「我的事项」操作）
22. **审核员处理客户反馈**（提意见→建跟进/外包修改/AI 草稿任务人工处理后重新发起确认；确认→进下一步）
23. **标记任务完成**（每条任务由审核员人工 completed 收口）
24. **形成首轮执行记录**（任务中心即记录：每步有任务、状态、意见、时间戳；另写一页复盘）
25. **进入下一阶段或暂停**（负责人决策——继续内容运营/小额测试，或先补能力）

## Chapter 6：Role-by-Role Operating Instructions（分角色操作说明）

### 客户账户 merchant
登录 → 自动进入**客户工作台**：项目进度、待配合事项、我的确认事项计数。
- 进「**我的事项**」（导航即有）→ 看待确认内容（确认什么 / 为什么 / 确认后下一步）。
- 内容没问题 → 点「**确认通过**」；需要调整 → 填修改意见（必填）→「提交修改意见」，等团队处理，**不需要重复提交**。
- 看不到内部任务、AI 草稿、外包信息（系统强制）；不直接调用 AI；不直接联系外包（除非负责人授权）。
- 记住：确认通过 = 同意进入下一步，**不代表承诺增长结果**。

### 采集员 collector
登录 → **采集员工作台**：我的任务实时计数 + 新建商家/接入向导入口。
- 进「任务中心」→ 看分配给自己的采集任务（要求 + 验收标准）。
- 线下用 **Field Pack** 采集原始资料 → 回系统按接入向导填商家基础信息 / Profile / Baseline / Operating Capacity。
- **每条信息标证据来源**；口述标口述；不知道写待补充。
- 任务详情「开始任务」→ 填成果摘要 →「提交审核」；被退回 → 看修改意见 → 补充 → 重新开始并提交。
- 不做最终诊断、不分配外包、不承诺结果。

### 审核员 operator（核心运营角色）
登录 → **人工审核工作台**：待审核 / AI 草稿 / 外包成果 / 待客户确认 / 客户已反馈 实时队列。
- 审核采集资料：对照证据来源，缺证据退回（意见必填），**没有证据不通过**。
- 用 **AI 工作台**：选商家选任务 → 复制 Prompt → 外部 AI → 粘贴 → 人工修改 →「提交为 AI 草稿审核任务」→ 审核 → **人工保存到节点**。
- 创建外包任务（要求+验收标准必填，不暴露经营数据）→ 分配 executor → 验收成果（通过/退回）。
- 创建客户确认事项（客户语言）→ 分配客户 → 发起 → 处理客户反馈 → 人工收口 completed。
- 红线：不绕过证据纪律；审核通过 ≠ 自动完成 ≠ 发布；AI 草稿必人审后人存。

### 外包 executor
登录 → **外包/执行工作台**：只显示分配给自己的任务计数。
- 进任务中心 → 任务详情：看**要求 + 验收标准 + 素材说明**。
- 完成后填「成果说明*（必填）/ 成果链接（云盘等外部地址）/ 补充备注」→「**提交成果，等待审核**」。
- 被退回 → 看修改意见 → 修改 → 重新提交（系统只保留最新提交）。
- 看不到商家完整经营数据、其它外包任务、内部审核内容（系统强制）；不直接联系客户（除非授权）；成果不含夸大/承诺文案、不侵权。

### 管理员 admin
登录 → **平台管理工作台**：全局任务统计、数据边界（DEMO/UAT/真实计数）。
- 只做：账号与角色管理、权限问题、异常处理、全局核查。
- **不作为日常运营账号**（日常审核用 operator）；**不多人共享**；角色变更=业务决策，按账号指南人工执行并记录。

## Chapter 7：AI Usage Runbook（AI 使用手册）

**AI 可用于（均为草稿，必经人审）**：资料缺口检查、商家诊断草稿、内容方向草稿、图片 brief、视频脚本、外包任务说明草稿、验收标准草稿、客户确认说明草稿、复盘草稿。

**AI 禁止用于**：自动承诺结果、自动修改商家节点、自动通过审核、自动发布、自动投流、自动给客户发送未审核内容、编造商家数据、把 UAT/DEMO 当真实案例。

**标准步骤（九步，与系统一致）**：
1. 审核员打开 AI 工作台（`/dashboard/ai-workbench`）
2. 选择商家和 AI 任务类型
3. 复制系统生成的 Prompt（已含商家上下文 + 安全规则，空字段标待补充）
4. 在外部 AI 工具生成草稿（当前不接 API，人工复制）
5. 粘贴回系统回填区
6. **人工修改**：删除编造内容、给无据判断标「待验证」、删除任何承诺表述
7. 点「提交为 AI 草稿审核任务」（进入待审核队列，提交 ≠ 通过）
8. 审核员对照验收标准审核（通过 / 退回重做）
9. 通过后**人工**保存到业务节点（详情页"去节点编辑页"按钮），或用于创建后续任务

> 系统保证：AI 输出永远不会自动写入业务节点（UAT 彩排实测断言）。

## Chapter 8：Outsource Runbook（外包手册）

- 外包任务**由审核员创建**（executor / merchant 不能建内部任务）。
- 外包**只看自己的任务**（系统强制：仅 assignedProfileId=本人的 outsource_execution 可见）。
- 任务**必须包含**：清晰要求 + 验收标准（+ 截止时间 + 素材说明）。创建页有五条提醒，照做。
- 如附 AI brief，**必须是人工审核后的内容**——未审核 AI 草稿不得直接给外包。
- 外包提交：成果说明（必填）/ 成果链接（云盘/图片/视频/文档外部地址）/ 备注。
- **审核员验收通过后**，才可进入客户确认环节。
- **通过 ≠ 发布**——是否对外使用由后续客户确认与负责人决定。
- 退回必附修改意见 → 外包重新提交（仅保留最新版）。
- **不自动结算外包费用**（结算线下人工，系统当前无结算功能）；**不自动触发客户确认**（由审核员人工创建）。

## Chapter 9：Client Confirmation Runbook（客户确认手册）

- 客户**只看到分配给自己的确认事项**（系统强制）。
- 内容必须**简单清楚**：确认什么 / 为什么确认 / 确认后进入哪一步——用客户语言，不用内部术语。
- **不暴露**：内部 AI Prompt、外包讨论与报价、审核争议、内部任务细节。
- 客户「确认通过」后：**不自动发布**、不自动完成——审核员看到"客户已确认"后人工标记完成并人工推进下一步。
- 客户提修改意见后：进入审核员处理——由审核员**决定**是否创建 外包修改任务 / AI 草稿任务 / 通用跟进任务（系统不自动创建）；处理完重新发起确认。
- 客户不需要重复提交意见；团队调整后会重新发起。
- 对客户重申：确认通过 = 流程放行，**不代表承诺增长结果**。

## Chapter 10：Data Discipline & Evidence Rules（数据与证据纪律）

依据证据宪法（E0–E3），首单执行中：

- **真实数据必须有来源**（E1 商家事实：谁说的/哪份单据/哪个后台；E3 系统事实：哪个系统读数）。
- 口述数据标「口述」；估计数据标「估算」；不确定标「待验证」。
- **不完整信息不能强行下结论**——诊断/方案里的缺口如实写"待补充"。
- **AI 输出 = E0 模型推测，不是事实**，禁入交付物当事实使用；必须人工核实或标待验证。
- **外包成果不能当最终成果**——必经审核验收。
- **客户确认 ≠ 增长结果**——只是流程放行。
- **不承诺 ROI / 客流 / 转化 / 销售额**（对内对外一致）。
- **DEMO / UAT 永不用作真实案例**、不进经验库、不对外引用。

## Chapter 11：Do / Do Not Checklist（一页清单）

| ✅ Do | ❌ Do Not |
|------|----------|
| 先授权再动手 | 未授权录入真实隐私 |
| 先采集再诊断 | AI 自动决策 / 自动写节点 |
| 先审核再进下一步 | 外包看完整经营数据 |
| 每条信息标注来源 | 客户看到内部 Prompt / 外包信息 |
| 用 AI 出草稿 | 未审核内容交给客户 |
| 人工审核每一份草稿/成果 | 未经客户确认就发布 |
| 外包严格按任务要求执行 | 承诺增长结果 |
| 客户确认通过后再推进 | 把 UAT / DEMO 当案例 |
| 每一步留任务记录（任务中心） | 多人共用 admin |

## Chapter 12：First Pilot Day Checklist（首日清单）

> 首日目标只有一个：**资料准确 + 流程跑通**。不求快，不求全。

- ☐ 确认项目负责人（人名、联系方式）
- ☐ 确认商家授权（Chapter 2 十条逐条核对）
- ☐ 确认账号就绪（collector / operator / executor / merchant 各可登录且工作台正确）
- ☐ 创建真实商家（真实名称，无前缀）

## Chapter 14：Risk Handling（常见风险与处理）

| 风险 | 处理方式 |
|------|---------|
| 商家资料不完整 | 标待补充照常推进可做部分；关键缺口（授权/基线）则停在采集环节，负责人跟商家约补充，不编不猜 |
| 商家不配合确认 | 确认事项停在待确认；负责人线下沟通；连续多次不配合 → 升级负责人评估是否暂停试点（商业决策） |
| AI 输出编造 | 人工修改阶段删除；审核阶段退回；把"编造样例"记入复盘，提示下次 Prompt 注意；**绝不带编造内容过审** |
| 外包成果不达标 | 退回 + 具体修改意见（必填）；两轮仍不达标 → 审核员换人/收回内部做（线下决策），任务取消由 admin 执行 |
| 客户频繁修改 | 每轮意见都记录在任务中；第三轮起负责人介入对齐期望；不在系统外私下改需求 |
| 账号权限配置错误 | admin 按账号指南核对角色 → 修正 → 用角色验证清单复测；错误期间产生的数据由 operator 复核 |
| 任务误分配 | operator/admin 重新分配（仅未开始/已分配状态可改）；已开始的先退回再重分配并备注 |
| 错把 UAT / DEMO 当真实 | 立即停止引用；检查名称前缀；向负责人报告；UAT/DEMO 数据不得出现在任何对客户/对外材料 |
| 未授权数据进入系统 | 立即停止录入；负责人评估删除（删除=人工决策，admin 执行）；复盘录入流程哪一步失守 |
| 有人试图承诺增长结果 | 当场纠正（对照 Outreach Kit 承诺边界）；已对客户说出口的由负责人出面更正预期；记入复盘 |

## Chapter 15：First Pilot Completion Criteria（首轮完成标准）

**首轮试运行完成 = 流程完成，不是增长结果。** 满足以下全部即可宣布首轮完成：

- ☐ 商家资料采集完成（含来源标注；缺口已列待补充清单）
- ☐ 采集审核完成（至少一次退回-重交往返更佳）
- ☐ AI 草稿审核完成（至少一份草稿走完 生成→人审→通过）
- ☐ 至少一个业务节点**人工保存**（如 TB-001 诊断）
- ☐ 至少一个外包任务完成（如本单需要外包）
- ☐ 至少一个客户确认事项完成（确认或意见往返均可）
- ☐ 任务记录完整（任务中心可回放每一步：状态/意见/时间）
- ☐ 风险与缺口记录清楚（复盘任务里写明）
- ☐ 全程**未承诺增长结果**
- ☐ **未发生数据泄露**（外包未见经营数据、客户未见内部内容、无密钥外泄）
- ☐ 形成首轮复盘记录（跑通了什么 / 卡点 / 系统增强建议 / 是否扩大）

## Chapter 16：Recommended Next Actions（下一步建议）

**结论：系统已具备首家真实商家试运行条件（UAT 彩排 PASS + 生产 RC Conditional Pass + 本 Runbook），但必须由人工商业负责人授权后才开始。**

建议顺序：
1. **准备真实分角色账号**（按 Chapter 3 + 内部账号指南；admin 人工执行；处理 admin@tot.local 夹具）。
2. **确认首家商家授权**（Chapter 2 十条逐条核对；用 Outreach Kit / Prospect Tracker 推进到正式授权）。
3. **按本 Runbook 进入系统**（Chapter 5 二十五步顺序执行；首日按 Chapter 12，首周按 Chapter 13）。
4. **首单只跑小范围闭环**（资料→诊断→一批内容/素材→客户确认；不放量、不投流、不承诺）。
5. **首单结束后复盘系统卡点**（按 Chapter 15 标准收口 + 复盘记录；把流程卡点转成系统增强清单）。
6. **再决定是否扩大试点**（商业决策：第二、三家进入 → 期间按"单商家闭环 → 3-5 家 → 再规模化"节奏，不提前平台化）。
