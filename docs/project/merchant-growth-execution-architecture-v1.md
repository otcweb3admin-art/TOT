## ⚠️ 两条最高约束（贯穿全文）

**① Evidence Framework**：所有执行记录必须可追溯，**执行不能只靠口头汇报**。关键执行动作必须留下：**执行时间 · 执行人/责任方 · 执行动作 · 执行产物 · 执行证据 · 偏差说明**。

**② Human Commercial Authority**：AI 可以**提醒执行/检查偏差/整理执行记录/识别执行风险/提供纠偏建议**；AI **不可**最终认定执行完成、不可决定更换执行人/追加资源/改变客户承诺/修改核心计划。**最终执行确认权属于：项目负责人 · 增长负责人 · 执行负责人 · 公司管理层。**

---

## 序言：执行是计划进入真实世界的落地层

> **执行不是"做了"，不是"发了内容"，不是"完成任务"。执行是：按照已确认计划，在规定时间内，由明确责任方，完成可检查动作，留下可追踪记录，产生可验证结果输入。**

如果执行失真：MVS 无法验证 → 归因无法成立 → 复盘会误判 → 经验库会被污染。因此**增长执行是 MGOS 从计划进入真实世界的关键落地层**。

## 第一章　Growth Execution Definition（增长执行定义）

### 1.1 执行不是把任务打勾

> 执行必须产生**可检查、可追踪、可验证**的现实动作。

### 1.2 执行的阶梯（六层区分）

| 层 | 含义 | 够不够 |
|----|------|--------|
| **计划 Plan** | 行动路径 | — |
| **任务 Task** | 具体动作单元 | — |
| **执行 Execution** | 真实做动作 | — |
| **完成 Done** | 动作做完了 | ❌ 可能没证据/跑题 |
| **有效执行 Effective** | 做完 + 符合计划 + 有产物 | ❌ 可能没留证据 |
| **可验证执行 Verifiable** | 有效执行 + 留下可追踪证据（能进 MVS）| ✅ **TOT 要的就是这个** |

```
做了 ──► 完成 ──► 有效 ──► 可验证   （只有最右端才算数）
```

> **"完成" ≠ "有效" ≠ "可验证"**：发了内容（完成）但跑题（无效）；有效但没留链接/没埋钩子（不可验证）。**只有可验证执行才能支撑 MVS 验证与归因。**

## 第二章　Plan-to-Execution Bridge（计划到执行的桥）

> **不得脱离计划自由执行**——脱离计划 = 失控。

执行必须承接计划（TASK-018）的输出：90 天计划 / 30 天行动计划 / 周计划 / 计划单元 / 目标 / 负责人 / 时间节点 / 所需资产 / 风险预案 / 验证节点。

| 计划输出 | 执行如何承接 |
|---------|-------------|
| 周计划 | → 本周执行清单（谁做什么）|
| 计划单元 | → 执行单元（一一对应落地）|
| 验证节点 | → 执行须埋的**证据钩子** |
| 责任人/时间 | → 执行的责任与 deadline |

### 2.1 想改计划 → 走调整机制，不在执行层擅改

> 如需调整计划，**必须回到计划调整机制**（018 第九章），**保护 MVS 归因有效性**。执行层的纪律 = **忠实于计划**——不在执行中私自改变量（换平台/加打法/加投流），否则破坏归因。

## 第三章　Execution Unit Model（执行单元模型）

每个执行单元至少包含：

| 字段 | 含义 |
|------|------|
| 执行目标 | 这个单元要达成什么 |
| 执行动作 | 具体做什么 |
| 执行责任方 | 谁做（内部/外包/AI 辅助）|
| 执行时间 | 何时做 |
| 执行前置条件 | 依赖什么就位 |
| 执行所需资产 | 用什么 |
| **执行产物** | 实际产出（内容/活动/动作）|
| **执行证据** | 可追踪记录（链接/截图/记录）|
| 执行状态 | 待执行/执行中/完成/验收/偏差/阻塞 |
| 偏差记录 | 与计划的差异 |
| 风险记录 | 执行中暴露的风险 |
| 验证关联 | 对应哪个验证节点 |

> **铁律：没有证据的执行，不得视为有效执行。** 执行单元是计划单元的"实际落地记录"——计划单元是"打算做"，执行单元是"真做了 + 产物 + 证据"。

## 第四章　Execution Responsibility Model（执行责任体系）

| 角色 | 执行职责 |
|------|---------|
| **AI（AIGO）** | 生成/辅助（内容/方案草稿、提醒、检查），**不认定完成** |
| **人工确认** | 审核草稿、验收执行 |
| **内部团队** | 线上执行主力（运营/内容统筹）|
| **外包团队** | 重执行（拍摄/剪辑/设计）|
| **商家配合** | 提供资料/授权/配合采集/经营反馈/**承接真实客户** |
| **项目负责人** | 执行验收、最终确认 |

### 4.1 商家不是主要执行方

> **TOT 公司与系统承担主要线上执行责任。** 商家主要负责：提供业务资料 · 提供授权 · 配合素材采集 · 提供经营反馈 · 承接真实客户。

> 这是"AI + 人 + 外包 + 商家"协作执行（承 commercial-operating）；**客户关系与验收在人**，AI 不认定完成（Human Commercial Authority）。

## 第五章　Content Execution Architecture（内容执行体系）

涵盖：素材采集 · 内容选题 · 脚本生成 · 图文/视频制作 · 发布节奏 · 平台适配 · 内容审核 · 发布记录 · 内容表现记录。

```
素材(TB-003) → 选题/脚本(AI草稿+人审) → 制作(人/外包) → 审核(禁区/合规/拦空话) → 发布(按节奏/平台适配) → 发布记录(链接+时间) → 表现记录(互动/效果)
```

> **内容执行不是创意发挥。内容执行必须服务于已确认增长策略与计划**（服从定位/栏目/比例/频率，承 TB-004）。
> - **跑题 = 无效执行**（发了但不服务策略）；内容审核拦截跑题/违规/空话。
> - **证据**：每条内容留发布链接 + 时间 + 表现数据（为复盘/归因）。
> - AI 生成选题/脚本草稿（ai_generated）→ 人审 → 制作；AI 不发布、不认定完成。

---

## 第六章　Channel Execution Architecture（渠道执行体系）

涵盖：账号建设 · Google Maps · TikTok · Facebook · Instagram · 小红书 · 私域 · 本地平台 · 平台规则遵守 · 渠道动作记录。

> **渠道执行必须留下可追踪来源，支持后续 MVS 归因。**

| 渠道 | 可追踪机制（归因钩子）|
|------|----------------------|
| Google Maps | 专属动作 / 搜索·导航来源 |
| 短视频（TikTok/小红书）| 专属链接 / 专属优惠码 |
| 私域 | 加好友来源标记 |
| 落地页 | 专属 URL |

> - **平台规则遵守**：渠道执行须合规（不刷评/不违规），否则封号风险 + 污染验证。
> - **渠道动作记录**：每个动作留时间/动作/责任方/可追踪标识。
> - **核心产物之一是"可追踪入口"**——这是 MVS 归因的物理基础。**没埋钩子的渠道执行 = 不可归因。**

---

## 第七章　Campaign Execution Architecture（增长活动执行体系）

涵盖：引流活动 · 转化活动 · 私域活动 · 优惠活动 · 口碑活动 · 投流测试（P-001）· 到店/咨询转化设计。

**每个 Campaign 必须说明**：

| 字段 | 内容 |
|------|------|
| 目标 / 人群 / 渠道 | 打谁、在哪打 |
| 动作 / 预算 / 时间 | 怎么打、花多少、何时 |
| **归因方式** | 专属码/链接/落地页（隔离自然客）|
| 验证方式 / 风险 | 怎么证明有效、风险预案 |

> - **Campaign 是最可归因的执行单元**（明确动作 + 预算 + 可追踪）。**投流 P-001** 尤其（花费 + 转化可追踪）：测试期小额 → 验证获客成本 → 放量/暂停；须隔离自然客（专属码/落地页）。
> - **先承接后引流**（承增长铁律）——Campaign 执行前确认承接能力，否则引流 = 浪费。
> - 没埋专属归因的活动，效果无法归因。

## 第八章　Execution Monitoring and Deviation Control（执行监督与偏差控制）

### 8.1 偏差分级

| 偏差级 | 含义 | 对验证的影响 | 处置 |
|--------|------|-------------|------|
| **轻微偏差** | 不影响计划目标与变量（如发布晚一天）| 无 | 记录，自行纠回 |
| **重大偏差** | 影响计划目标/资源（如核心动作没做、产能不足断更）| 可能 | **人工确认** + 纠偏 |
| **破坏验证的偏差** | 改变了关键变量（擅自换平台/加投流/改打法）| **破坏归因** | **必须回到计划调整机制**（018 Ch9），重设归因基线 |

### 8.2 偏差不是失败，未记录的偏差才是风险

> - **偏差不是失败**——记录的偏差可纠、可解释。
> - **未记录的偏差才是风险**——让复盘误判（以为按计划做了，其实没有）。
> - 监督机制：周计划"是否偏离"检查（承 018 Ch7）→ 早发现。**AI 检查偏差 + 提示，人决定处置。**
> - 偏差控制的核心是**保护归因**：破坏变量的偏差必须显式处理，不能悄悄发生。

## 第九章　Execution Evidence and Record（执行证据体系）

执行记录服务于：**MVS 验证 · KPI 复盘 · ROI 计算 · 经验沉淀 · 客户汇报**。

### 9.1 必须保留的执行证据

```
发布记录 · 内容链接 · 活动记录 · 优惠码 · 私信截图 · 咨询记录 · 到店来源 · 投流记录 · 预算使用记录 · 商家反馈
```

### 9.2 执行证据是 MVS 的前提

> - **执行不能只靠口头汇报**（"我发了""效果不错"）——必须留可追溯证据（链接/截图/记录），能答证据五问。
> - 执行证据多为 **E1**（真实记录），须真实、**及时**（每周记录，承 018）、可追溯。
> - **"做了但没证据"在系统里 = 没做**（不可验证执行不算数）——这逼出"**边做边留证据**"的纪律。

## 第十章　Execution Quality Gate（执行质量门）

一项执行进入"完成"状态前**必须满足**：

```
☑ 动作真实发生   ☑ 责任方明确   ☑ 时间清楚   ☑ 产物存在   ☑ 证据留存
☑ 偏差已记录     ☑ 风险已说明   ☑ 与计划一致或有调整依据   ☑ 与验证节点关联   ☑ 人工确认完成
```

> 不满足 → **不得进入验证阶段**。这是 MGOS **第四道质量门**（诊断 1 / 策略 2 / 计划 3 / 执行 4）。

**防"假完成"**：打了勾但没产物/没证据/跑了题 = 假完成。质量门拦截假完成进入验证——否则**验证的是假数据 → 污染**。

---

## 第十一章　Human Execution Review Principle（人工执行确认原则）

> **执行结果必须经过人工确认。**

| AI 负责 | 人负责 |
|---------|--------|
| 汇总执行记录、检查完整性、识别偏差、提示风险、提供纠偏建议 | **确认是否有效完成、是否补做、是否进入验证、是否调整计划、承担最终责任** |

- **AI 不可**：最终认定执行完成、更换执行人、追加资源、改客户承诺、改核心计划。
- **最终执行确认权**：项目负责人 / 增长负责人 / 执行负责人 / 公司管理层。
- 与 AIGO + Human Commercial Authority 一致：**AI 整理证据，人拍板**。

### 12.1 执行偏差如何防止污染经验库

> 偏差记录 + 执行质量门 → **只有"可验证执行"进验证 → 只有验证过的进经验** → 防污染。
> **核心警示**：执行失真 → MVS 无法验证 → 归因不成立 → 复盘误判 → 经验污染。

## 第十三章　Architecture Conclusions（架构结论）

1. **执行是"可验证执行"**：真实动作 + 符合计划 + 有产物 + 留可追踪证据；打勾不算。
2. **执行必须忠实于计划**：不在执行层擅改变量；要改走计划调整机制（保护归因）。
3. **执行单元须带产物与证据**：没有证据的执行不算有效执行。
4. **TOT 承担主要线上执行责任，商家配合**；AI 辅助、人确认。
5. **内容执行服务策略**（不是创意发挥）；渠道执行**必埋可追踪入口**；活动执行**必带专属归因**。
6. **偏差分级**：轻微（纠回）/ 重大（人确认）/ 破坏验证（回调整机制重设基线）。
7. **偏差不是失败，未记录的偏差才是风险**。
8. **执行证据是 MVS 的前提**：口头不算数，边做边留证据，"做了没证据=没做"。
9. **执行质量门**（十项）拦假完成，不达标不进验证——MGOS 第四道质量门。
10. **AI 整理证据、人确认完成**：AI 不认定完成/不换人/不追加资源/不改承诺/不改核心计划。

### 13.1 六个根本问题的回答

| 问题 | 回答 |
|------|------|
| **为什么执行不是任务打勾？** | 打勾 ≠ 有效 ≠ 可验证；执行要真实动作 + 可追踪证据。|
| **为什么执行必须可追踪？** | 不可追踪 = 无法验证、无法归因。|
| **为什么执行证据是 MVS 的前提？** | MVS 靠真实数据 + 可追踪来源归因；无证据无从验证。|
| **为什么偏差必须记录？** | 未记录的偏差让复盘误判（以为按计划做了，其实没有）；记录的可纠可解释。|
| **为什么没有执行质量门就无法验证增长？** | 假完成（无产物/无证据/跑题）进验证 = 验证假数据。|
| **为什么执行失真会污染经验库？** | 假执行→假结果，若当成功沉淀=假经验；偏差乱改变量→归因失真→错经验。|

### 13.2 最终结论

> **增长执行是 MGOS 从计划进入真实世界的关键落地层。** 执行 = **可验证执行**（真实动作 + 可追踪证据）——它**边做边留证据、偏差必记录、质量门拦假完成、人确认完成**。**执行证据是 MVS 的前提，执行质量直接决定验证与经验的真伪。** 执行真，增长才可被证明；执行假，全链皆假。
