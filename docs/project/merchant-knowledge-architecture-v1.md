## 0. 立论：TOT 不是营销建议生成器，而是「增长知识引擎」

TOT 的目标不是"生成营销建议"，而是**提升真实商家的增长成功率**。要做到这一点，系统必须把每一次代运营沉淀为**可复用、可验证、可累积的知识**，而不是一次性的方案文本。

因此 TB-001~008 的本质不是一条"工作流"，而是一条 **知识生产流水线**，驱动一个永不停止的核心闭环：

```
        ┌──────────────────────────────────────────────────────┐
        │                                                      │
        ▼                                                      │
   Knowledge ──► Asset ──► Execution ──► Result ──► Experience ─┘
   （知道）      （资产）    （执行）      （结果）     （经验）
     ▲                                                  │
     └────────── 经验回流，提升下一轮的"知道" ───────────┘
```

- **Knowledge（知道）**：关于商家、市场、竞品、平台、增长机制的结构化认知。
- **Asset（资产）**：把知识物化为商家的线上资产（账号、内容、活动）。
- **Execution（执行）**：真实做了什么、何时做、花了多少。
- **Result（结果）**：真实客流、询盘、转化、ROI。
- **Experience（经验）**：从"方案→执行→结果"中提炼出的"条件→动作→结果"模式。
- **回流**：经验成为新的知识，让下一个商家少走弯路。

> **AI 在这条链上的身份不是决策者，而是证据工作者**：研究员、分析师、方案工程师、审计员。决策权恒在人工。整个系统遵循 **Evidence-Based Growth（基于证据的增长）** 原则——**没有证据的判断，不得进入交付物，更不得进入经验库。**

## 1. Knowledge Taxonomy（知识分类体系）

系统需要掌握 **7 大类知识**。每一类用四个维度刻画：**作用范围**（私有=单商家 / 共享=可跨商家复用）、**变化速度**（稳定/慢变/快变）、**主要来源**、**对应 TB**。

### 1.1 商家知识 Merchant Knowledge　【私有 · 稳定】
- **子类**：身份（行业/门店/规模）、客群画像、核心卖点、**执行红线**、预算与**投流意愿(P-001)**、增长目标、线上成熟度、增长前基准、合作意向、执行能力边界。
- **特征**：单商家私有；相对稳定（一轮交付内基本不变）；是全链路的**约束基准**。
- **来源 / TB**：商家提供 / **TB-001**。

### 1.2 市场知识 Market Knowledge　【半共享（按区域×行业）· 慢变】
- **子类**：区域市场（如某城市华人区）、本地消费能力、行业趋势、季节性、需求结构、**竞争强度**。
- **特征**：可被同区域/同行业的多个商家共享；季度级慢变；需要外部调研。
- **来源 / TB**：外部（网络、行业报告、本地观察）/ 贯穿 TB-001 诊断与 TB-006/008 策略。

### 1.3 竞品知识 Competitor Knowledge　【半共享（按行业×区域）· 快变】
- **子类**：竞品账号（定位/粉丝/活跃度）、竞品内容（选题/形式/爆款）、竞品活动、**竞品投流**（平台/节奏/钩子）。
- **特征**：同行业/区域共享；快变（账号内容每周变）；**保留快照**而非长期持有。
- **来源 / TB**：外部研究 / 支撑 TB-002/004/006 的差异化与引流设计。

### 1.4 平台知识 Platform Knowledge　【全局共享 · 快变】
- **子类**：各平台机制——TikTok / Facebook / Instagram / Google Maps / 私域；**规则与增长机制**（推荐逻辑、合规红线、本地 SEO、广告政策、内容规范、冷启动规律）。
- **特征**：全体商家共享（平台规则对所有人一样）；快变（平台政策频繁调整）；**强时效，须带时间戳**。
- **来源 / TB**：外部（平台官方规则/文档/实测）/ 支撑 TB-002~006 的所有平台动作。

### 1.5 增长机制知识 Growth-Mechanism Knowledge　【全局共享 · 慢变】
- **子类**：通用增长原理——"先承接后引流""先信任后转化""先稳定后放大""内容/营业系统优先于增长系统"、冷启动四阶段、降级原则、归因方法（可追踪来源法）。
- **特征**：方法论层面，跨平台跨行业；慢变；是模板"AI 生成要点/审核要点"的底层逻辑。
- **来源 / TB**：沉淀于 **Knowledge（模板库）本身** + 经验回流 / 贯穿全链。

### 1.6 执行知识 Execution Knowledge　【私有 · 流水（事件型）】
- **子类**：**做了什么**（哪个动作/方案落地）、**什么时候做**（时间）、**花费多少**（人力/预算）、由谁承接（人/外包/AI）、实际 vs 计划的偏差。
- **特征**：单商家私有；事件流水（持续产生）；是"方案→结果"之间的**因果桥梁**——没有执行记录，结果就无法归因。
- **来源 / TB**：执行阶段记录 / **TB-002~TB-006 的实际执行**（计划来自方案，执行是真实动作）。

### 1.7 结果知识 Result Knowledge　【私有 · 周期性事实】
- **子类**：客流、询盘/咨询量、转化率、营业额、粉丝/私域增长、内容效果、直播效果、**投流 ROI 与获客成本(P-001)**、可追踪线上来源客流（归因）。
- **特征**：单商家私有；**真实事实**（非方案数据）；周期采集；是复盘与分成的唯一依据。
- **来源 / TB**：真实经营数据（人工/POS/平台）录入 / **TB-007 Input**。

### 1.8 经验知识 Experience Knowledge　【共享（去标识后）· 累积型 · 最高价值】
- **子类**：**成功模式**、**失败模式**、**适用条件**（在什么行业/城市/渠道/预算下，什么动作带来什么结果）。
- **特征**：由私有知识**去标识化提炼**而来，转为可跨商家复用；只增不减（累积）；是系统的**护城河**。
- **来源 / TB**：**TB-007 跨商家提炼** + 闭环回流 / 反哺 TB-001 诊断与 TB-008 策略。

### 1.9 分类总览表

| 知识类 | 范围 | 变化 | 证据归属 | 主要来源 | 对应 TB |
|--------|------|------|---------|---------|---------|
| 商家知识 | 私有 | 稳定 | E1 | 商家 | TB-001 |
| 市场知识 | 半共享 | 慢变 | E2 | 外部调研 | TB-001/006/008 |
| 竞品知识 | 半共享 | 快变 | E2 | 外部研究 | TB-002/004/006 |
| 平台知识 | 全局 | 快变 | E2 | 平台官方 | TB-002~006 |
| 增长机制 | 全局 | 慢变 | E2/E3 | 模板+经验 | 全链 |
| 执行知识 | 私有 | 流水 | E1 | 执行记录 | TB-002~006 |
| 结果知识 | 私有 | 周期 | E1 | 真实数据 | TB-007 |
| 经验知识 | 共享(去标识) | 累积 | **E3** | 跨商家提炼 | TB-007→全链 |

## 2. Knowledge Source Map（知识来源图）

**每类知识从哪里来 + 谁负责 + 信任级别 + 刷新频率**。核心区分：**外部知识 / 商家知识 / 执行知识 / 结果知识 / 战略知识 / 经验知识**。

```
┌────────────────────────── 知识来源全景 ──────────────────────────┐
│                                                                   │
│  【外部世界】                          【单个商家交付链】           │
│  ┌─────────────────┐                  ┌────────────────────────┐   │
│  │ 网络 / 行业报告  │── 市场知识 ──┐    │ TB-001 ── 商家知识(E1)  │   │
│  │ 竞品账号 / 内容  │── 竞品知识 ──┤    │ （行业/客群/卖点/红线/  │   │
│  │ 平台规则 / 政策  │── 平台知识 ──┤    │   预算/目标，商家提供）  │   │
│  └─────────────────┘             │    ├────────────────────────┤   │
│     由 AI-研究员采集               ├──► │ TB-002~006 ─ 执行知识   │   │
│     （E2，须多源佐证、带时间戳）    │    │ （做了什么/何时/花多少） │   │
│                                   │    ├────────────────────────┤   │
│                                   │    │ TB-007 ── 结果知识(E1)  │   │
│  【系统自身】                      │    │ （客流/询盘/转化/ROI，  │   │
│  ┌─────────────────┐             │    │   真实数据，非方案数据） │   │
│  │ 模板库(TB规则)   │─ 增长机制 ──┤    ├────────────────────────┤   │
│  │ 历史案例库       │─ 经验知识 ──┘    │ TB-008 ── 战略知识      │   │
│  └─────────────────┘  (E3)            │ （90天路线/KPI/下一轮）  │   │
│        ▲                               └───────────┬────────────┘   │
│        │   去标识化提炼(TB-007 跨商家)              │                │
│        └────────────────────────────────────────────┘                │
│                     经验回流（Experience → Knowledge）                 │
└───────────────────────────────────────────────────────────────────┘
```

| 来源域 | 产出知识 | 负责人（业务角色）| 信任默认级 | 刷新频率 |
|--------|---------|------------------|-----------|---------|
| 网络/行业报告 | 市场知识 | AI 研究员 + 人工复核 | E2 | 季度 |
| 竞品账号/内容 | 竞品知识 | AI 研究员 | E2（快衰减）| 周/月 |
| 平台官方规则 | 平台知识 | AI 研究员 + 人工 | E2（须时间戳）| 事件驱动 |
| 商家提供(TB-001) | 商家知识 | 采集员 + 商家确认 | E1 | 一轮一次 |
| 执行记录(TB-002~006) | 执行知识 | 执行/运营/外包 | E1 | 持续 |
| 真实数据(TB-007) | 结果知识 | 商家/运营录入 | E1 | 周期 |
| 90天规划(TB-008) | 战略知识 | 管理员确认 | 派生(E1+E3) | 每周期 |
| 跨商家提炼(TB-007) | 经验知识 | AI 分析师 + 人工审定 | **E3** | 累积 |

**关键原则**：
- **商家知识必须由商家确认（E1）**，AI 不得替商家"脑补"画像或红线。
- **外部知识默认是 E2，不是 E1**——网络信息须多源佐证、标注来源与时间，否则降级为 E0。
- **经验知识是唯一的 E3**，且只能由"足够多的真实结果"提炼，不能由单个案例或推测生成。

## 3. Evidence Framework（证据体系 V1）

> 这是整个架构的**质量闸门**。Evidence-Based Growth 的落地点：**每一条进入交付物或经验库的知识，都必须携带证据标签。**

### 3.1 证据分级

| 级别 | 名称 | 含义 | 来源 | 验证方式 | 可否进交付物 | 可否进经验库 |
|------|------|------|------|---------|:-----------:|:-----------:|
| **E0** | 模型猜测 | AI 未经证据的推断/脑补/常识填充 | AI 自身 | 无 | **禁止** | **禁止** |
| **E1** | 商家事实 | 商家提供并确认的真实情况 | TB-001 / 真实数据录入 | 商家/采集员确认 | ✅ | ✅（作原始证据）|
| **E2** | 市场事实 | 外部可查证的市场/竞品/平台事实 | 网络调研 / 平台官方 | **多源佐证 + 时间戳 + 人工复核** | ✅（须标来源）| 条件允许（佐证充分）|
| **E3** | 系统事实 | 系统历史真实案例提炼的模式 | 历史案例库 | **样本量 + 一致性 + 人工审定** | ✅（最高可信）| ✅（这是经验库主体）|

### 3.2 证据标签（每条知识都要带）

```
知识原子 = {
  内容,
  证据级别 ∈ {E0,E1,E2,E3},
  来源,            // 谁说的 / 哪查的
  时间戳,          // 何时为真（E2/E3 必填）
  置信度,          // 高/中/低
  样本量,          // E3 模式必填：基于多少真实案例
  适用边界         // 在什么条件下成立
}
```

### 3.3 进入经验库的准入规则

**允许进入经验库**：
- ✅ **E3 模式**：达到最小样本量 + 结果一致性 + 人工审定的"条件→动作→结果"模式。
- ✅ **E1 原始事实**：去标识后作为支撑 E3 的证据案例保留。
- ✅ **E2 佐证充分的市场/平台事实**：多源一致、人工复核通过，作为背景知识（须可被时效淘汰）。

**禁止进入经验库**：
- ❌ **任何 E0**（模型猜测）——一票拒绝，连交付物都不能进。
- ❌ **单样本"经验"**：只有 1 个案例不构成模式（是案例，不是经验）。
- ❌ **未确认的 E1**：商家未确认的"事实"。
- ❌ **无佐证/无时间戳的 E2**：孤证或来源不明的网络信息。
- ❌ **结果数据缺失的"成功经验"**：没有真实结果支撑的方案不得被标为"成功模式"。

### 3.4 与现有治理的咬合
- E0 禁入 = 各模板"**禁止空话/禁止能力幻想/禁止脑补**"的统一底层表达。
- E1/E2/E3 分级 = 各模板"**基于真实数据、禁止用方案数据代替实际数据**"的形式化。
- 准入规则 = REVIEW_POLICY"AI 只生成、人工审核"的延伸：**证据定级由 AI 标注，准入由人工裁定。**

## 4. Experience Accumulation Model（经验沉淀模型）

> 回答 Q4：TB-007 如何把一次交付沉淀为经验。核心转化：**方案 → 执行 → 结果 → 经验。**

### 4.1 从"一次交付"到"一条经验"

```
方案(Plan)         执行(Execution)        结果(Result)         经验(Experience)
TB-002~006 的      实际做了什么/          TB-007 真实           条件→动作→结果
Deliverable    ──► 何时/花多少      ──►   客流/转化/ROI    ──►  的可复用模式
（计划，E1派生）    （执行知识，E1）        （结果知识，E1）       （E3，须达阈值）

   单条三元组 = 一个【案例 Case】（私有，去标识后保留）
   N 个相似上下文的案例 + 结果一致 ──► 提炼为一条【经验 Pattern】（共享，E3）
```

### 4.2 经验原子结构

一条"经验"= 带证据的因果模式：

```
Experience = {
  Context（适用条件）: 行业 × 城市/区域 × 商家成熟度 × 预算档 × 平台 × 客群,
  Action（做了什么）  : 方案类型 / 动作组合 / 节奏,
  Outcome（结果）     : 关键指标变化 + 成功/失败标记,
  Evidence（证据）    : 样本量 N + 结果一致性 + 置信度 + 来源案例,
  Validity（有效边界）: 在何条件下成立 / 何时可能失效（如平台规则变）
}
```

**成功模式 / 失败模式 / 适用条件**三者一体：失败模式与成功模式**同等重要**（"在 X 条件下做 Y 会失败"是高价值经验）。

### 4.3 转化机制（TB-007 的经验生产职责）

1. **对标**：方案（计划）vs 执行（实际）vs 结果（真实）三方对比 → 得出"完成度 + 因果判断"。
2. **归因**：区分"执行问题"（动作没做到）与"策略问题"（做到了仍无效）——这是 TB-007 已设计的核心，也是经验真伪的关键。
3. **去标识 + 打标签**：剥离商家身份，给案例打上 Context 维度标签（行业/城市/渠道/内容类型/预算）。
4. **入案例库**：每个交付周期产出若干 Case（私有证据级 E1，去标识后可共享）。
5. **模式结晶**：当某 Context×Action 切片的案例数达阈值且结果一致 → 经人工审定，升级为 E3 经验。

### 4.4 永久保留 vs 仅快照

| 数据 | 处置 | 理由 |
|------|------|------|
| **E3 经验模式** | **永久** | 系统核心资产，常青 |
| **去标识案例三元组**（Context→Action→Outcome+证据）| **永久** | 经验的证据基底，支撑可追溯与再统计 |
| **结构性结果事实**（行业/区域基准、归因结果）| **永久（聚合）** | 支撑市场知识与基准 |
| 原始执行流水（逐条动作/日志）| **快照后归档/聚合** | 量大、价值随时间衰减，留聚合即可 |
| 平台规则、竞品具体内容（E2 快变）| **带时间戳快照** | 强时效，会被新版本取代，留历史用于解释当时判断 |
| AI 草稿/中间推理（E0）| **不留存为知识** | 猜测不是知识；仅作过程留痕，不入库 |
| 商家可识别身份信息 | **隔离，不入共享经验** | 隐私 + 合规；经验库一律去标识 |

**一句话**：**经验与去标识案例永久保留；时效性与原始流水只留快照；E0 一律不进知识库。**

## 5. Growth Intelligence Model（增长智能模型）

> 回答 Q5：系统如何从 100 家成长到 1000 家，并形成行业/城市/渠道/内容/投流经验。

### 5.1 飞轮（为什么会越用越强）

```
        每个商家走完一轮 = 贡献 N 条带维度标签的真实案例
                    │
                    ▼
   案例在各维度切片上累积 → 切片达到统计质量 → 结晶为高置信经验(E3)
                    │
                    ▼
   新商家冷启动时，按其 Context（行业×城市×渠道）召回已验证经验 → 少走弯路
                    │
                    ▼
   成功率提升 → 更多商家 → 更多案例 → 更密的切片 → 更强的经验  ↺（复利）
```

这就是护城河：**别人有模板，TOT 有"被真实结果验证过的、按维度切片的经验库"**。

### 5.2 五个经验维度（必须从第一天就给每个案例打标签）

| 经验维度 | 切片键 | 回答的问题 | 喂养的环节 |
|---------|--------|-----------|-----------|
| **行业经验** | 行业类型 | 这个行业什么打法有效/无效 | TB-001 诊断、TB-002 定位 |
| **城市/区域经验** | 城市×华人区 | 本地市场/消费/竞争特征 | TB-001 市场判断、TB-006 本地引流 |
| **渠道经验** | 平台 | 各平台冷启动/增长规律 | TB-002 平台选择、TB-004/005 |
| **内容经验** | 内容类型/选题 | 什么内容带来信任/转化 | TB-003/004 内容设计 |
| **投流经验** | 平台×预算档×客群 | 什么投流结构 ROI 最优(P-001) | TB-006 投流、TB-008 三级分支 |

> **架构铁律**：聚合维度必须**前置设计**——每个 Case 在产生时就带齐 `行业/城市/渠道/内容类型/预算档` 标签。否则到 1000 家时无法回切片，经验库形同散沙。

### 5.3 成长阶段（智能随规模升级）

| 规模 | 经验状态 | 系统能力 | 主导证据 |
|------|---------|---------|---------|
| **1–10 家** | 个案，无模式 | 靠外部知识(E2)+商家事实(E1)+模板机制 起步 | E1/E2 |
| **10–100 家** | 行业/渠道切片出现弱模式 | 同行业可给"参考打法"，置信中 | E2 + 初步 E3 |
| **100–1000 家** | 多维切片密集、模式高置信 | 新商家可按 行业×城市×渠道 召回"已验证打法"做冷启动 | **E3 主导** |
| **1000+ 家** | Growth Intelligence 成为独立资产 | 诊断更准、方案更稳、失败模式提前预警 | E3 网络效应 |

### 5.4 不变的约束
- 经验**永远去标识**，不串商家隐私（呼应不提前平台化/合规）。
- 经验**有有效边界与时效**：平台规则变 → 相关投流/渠道经验需重新验证，不可盲信旧经验。
- 经验是**建议来源**，不是自动决策来源——召回的经验仍要经人工判断是否适用本商家。

## 6. AI Responsibility Boundary（AI 职责边界）

> 回答 Q6：AI 是证据工作者，不是决策者。**禁止任何"AI 拍板"设计。** 四个职能：

### 6.1 四职能定义

| 职能 | 中文 | 能做（✅）| 不能做（❌）|
|------|------|----------|------------|
| **Researcher** | 研究员 | 采集外部知识（市场/竞品/平台），标来源、定 E2、加时间戳、多源佐证 | 把孤证或推测当事实；把 E2 谎报为 E1 |
| **Analyst** | 分析师 | 分析商家×市场×结果，找差距、做归因、提假设、跨案例找模式 | 替商家下结论；把假设当定论；越过证据级别断言 |
| **Planner** | 方案工程师 | 基于证据生成方案草稿（恒为 ai_generated），带证据标签+置信度+**降级方案** | 审批/放行/上线/花预算；生成无证据支撑的方案；违反商家红线 |
| **Auditor** | 审计员 | 审自己/他人产出：查 E0、查空话、查红线冲突、查证据完整性、查一致性 | 做**最终**通过裁定（终审是人工）；放行未达证据标准的内容入经验库 |

### 6.2 AI 总体能力边界

**AI 可以做**：
- 采集、结构化、**给知识定证据级别**（E0–E3）并打标签。
- 诊断、对比、归因、找差距、找模式、提优先级与合理化建议。
- 生成**基于证据**的方案草稿（含主方案+降级方案+风险四要素）。
- 审计产出：识别脑补(E0)、空话、能力幻想、红线冲突、参数缺失。
- 暴露不确定性：明确告诉人工"哪里是事实、哪里是推测、置信多少"。

**AI 不能做**：
- ❌ **拍板决策 / 最终审批 / 放行进入下一阶段**（终审恒在 admin）。
- ❌ 把 **E0 猜测**当作 E1/E2/E3 事实呈现。
- ❌ 让未达证据标准的知识进入**交付物或经验库**。
- ❌ 跳过人工审核、自动执行、花费预算、发布内容。
- ❌ 越过/改写**商家执行红线**与 **P-001 投流意愿**。
- ❌ 将单案例标为"成功/失败经验"（未达样本阈值）。

### 6.3 与现有治理的一致性
- 完全继承 REVIEW_POLICY（AI 只生成、人工审核）与 AI_WORK_RULES（Claude 仅建议权、无决策权）。
- **Evidence Framework 是这条边界的"技术化语言"**：AI 的产物必须可被证据级别检验，人工据此裁定。
- 一句话定位：**AI 负责"让证据充分、让判断就绪"；人负责"拍板"。**

## 7. Architecture Conclusions（架构结论）

1. **TOT 的真正产品是"增长知识引擎"，不是方案生成器。** 价值在 `Knowledge→Asset→Execution→Result→Experience→Knowledge` 闭环的复利，而非单次输出。

2. **知识分 7 类、3 个范围（私有/半共享/全局）、按变化速度分层管理。** 私有知识（商家/执行/结果）是燃料，经验知识（去标识后共享）是护城河。

3. **Evidence Framework（E0–E3）是系统的质量宪法。** 每条知识带证据标签；E0 禁入交付物与经验库；只有达阈值的 E3 模式才是"经验"。这把所有模板里分散的"禁空话/禁脑补/基于真实数据"统一成一套可执行规则。

4. **经验沉淀 = 方案→执行→结果→经验 的去标识化结晶。** 永久保留经验模式与去标识案例；时效性与原始流水只留快照；失败模式与成功模式同等珍贵。

5. **聚合维度（行业/城市/渠道/内容/投流）必须从第一个案例就打标签。** 这是从 100 家走向 1000 家、形成 Growth Intelligence 的前置条件——晚打标签 = 经验库报废。

6. **AI 是四职能证据工作者（研究/分析/方案/审计），永不拍板。** 决策权恒在人工；AI 的职责是把证据做足、把不确定性讲清。

7. **本蓝图是后续工作的统一上位依据**，指导但不替代：

| 后续工作 | 本蓝图给它的约束 |
|---------|-----------------|
| **Merchant Asset Catalog** | 资产必须可追溯到知识来源与证据级别 |
| **Evidence Framework（落地）** | E0–E3 标签 + 准入规则 + 去标识规则 |
| **Database Design** | 须能承载证据标签、聚合维度标签、案例/经验分离、快照 vs 永久 |
| **Workflow Design** | 每个 TB 节点既是"作业"也是"知识生产 + 证据定级"环节 |
| **AI Agent Design** | 四职能边界 + "证据足够才产出、人工才决策" |

> **成功标准达成自检**：本文不是技术方案，而是一份回答"**系统知道什么 / 如何知道 / 如何验证 / 如何成长**"的知识架构蓝图，可指导后续全部设计工作。
