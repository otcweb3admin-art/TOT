# SYSTEM_BLUEPRINT.md — 系统蓝图规划

> 本文件是「海外华人实体商家线上增长交付系统」的**系统总纲**。
> 上承商业模式，下接 8 个封板模板（TB-001~TB-008）与治理/架构文档，
> 是后续搭系统、组团队、对接外包、安排实施的总依据。
>
> **核心策略：完整骨架优先（宽而浅）** —— 先把整条流水线的流程框架全部搭起来、
> 让一个商家能从头到尾走通，每个流程先做"能跑就行"的简化版，精细打磨留到后面。

## 2. 业务模式

**一句话**：组建团队（含外包 + AI），用一套标准化、模块化的流程，批量为海外华人实体商家提供线上代运营服务，按客流量分成获利。

| 维度 | 内容 |
|------|------|
| 服务对象 | 海外华人实体商家——老字号（有产品力但不懂线上）+ 新店（缺线上启动能力），共同痛点是"线上营销短板" |
| 服务内容 | 全链路线上代运营：诊断 → 账号搭建 → 素材 → 内容 → 直播 → 引流 → 复盘 → 90天规划（即 TB-001~008） |
| 交付方式 | **流水线协作**：每个流程是一个模块，由"人工 / 外包 / AI"之一承接，做完人工审批，流转下一环 |
| 收费模式 | **按客流量分成**（初步设定方向；具体客流统计与归因口径见第 8 节，后续细化） |
| 价值主张 | 商家无需自建线上团队、不必懂线上，由专业流水线代运营，**按效果（流量）付费**，低门槛、见效快 |
| 团队构成 | 内部团队（采集/运营/管理）+ 外包（拍摄/剪辑等重执行环节）+ AI（生成各模板草稿） |

## 3. 系统定位与边界

**系统是什么**：支撑代运营流水线运转的「**流程协作系统**」——带权限、能流转、多角色协作、多商家并行的内部工作台。

| 系统是什么 | 系统不是什么 |
|-----------|-------------|
| 流程驱动的协作工作台（任务分派 + 审批 + 流转 + 状态管理）| 不是 BI / 数据分析系统 |
| 多角色、分权限的内部系统 | 不是 CRM / 财务结算系统（分成结算是独立模块，初期可手工）|
| 承载 8 模板作业流程的"流水线骨架" | 不是自动发布社交媒体的工具 |
| 人/外包/AI 灵活承接每个流程 | 不是无人工干预的全自动 AI（**所有 AI 产物必须人工审核**）|

**核心价值**：让流程**流水线化运转**——省人工、降成本、缩短运作时间、快速见效；让团队/外包/AI 各就各位、协同交付。

**边界原则**（呼应 SYSTEM_BOUNDARY / CODE）：先内部工具、先单商家闭环、**不提前平台化**（不做多租户/开放API/自动结算）。

## 4. 角色与权限

**角色清单**（基于 ROLE_MODEL，补充外包角色）：

| 角色 | 登录后看到 | 主要职责 | 负责的流程（示例）|
|------|-----------|---------|-----------------|
| **商家 merchant** | 自己的信息表、待确认的交付物 | 提供基础信息、确认/反馈交付结果 | TB-001 信息提供、各阶段交付确认 |
| **采集员 collector** | 待采集的商家任务 | 采集并录入商家信息 | TB-001 信息采集录入 |
| **运营 operator** | 分派给自己的运营任务 | 执行账号/内容/直播/引流等运营工作 | TB-002~006 运营类工作 |
| **执行 executor** | 分派给自己的执行任务 | 落地执行（拍摄/剪辑/发布等，常外包）| TB-003 素材拍摄、内容制作 |
| **外包团队 outsource** | 仅自己承接的外包任务 | 承接特定重执行模块，按规范交付 | TB-003 拍摄/剪辑等 |
| **管理员 admin** | 全流程总览、待审批任务 | 审批、调度、分派、退回争议裁决 | 各流程审批卡点、调度 |
| **AI worker** | （无登录，系统调用）| 按模板"AI生成要点"生成各产物草稿 | 各模板 AI 草稿生成 |

**权限设计原则**：
- 每个角色**只看到与自己相关的任务**（采集员看不到引流方案，外包只看到自己那段）
- **采用两级审批机制**（详见第 5 节）：①下阶段初审——下一阶段接收方依据上阶段交付成果人工验收；②管理员终审——初审通过后管理员最终放行。AI 和外包**无审批权**（只生成/执行）
- 角色与流程是"**分派关系**"，不是固定绑定——同一流程可由人/外包/AI 灵活承接（见第 6 节）

## 5. 流程流转设计

### 5.1 主流程链路（9 节点，对接 WORKFLOW_OVERVIEW + 模板链路）

```
商家接入
  → TB-001 商家诊断
  → TB-002 账号搭建
  → TB-003 素材采集
  → TB-004 内容运营
  → TB-005 直播规划
  → TB-006 引流转化
  → TB-007 数据复盘
  → TB-008 90天增长
  ↺ 循环回 TB-007（新一轮增长周期）或结束交付
```

每个节点 = 一个流程 = 对应一个 TB 模板（作业书）= 由一个角色/外包/AI 承接。

### 5.2 单个流程内部状态（对接 WORKFLOW_STATE_MODEL）

```
draft（草稿）→ submitted（提交）→ ai_generated（AI生成）
  → reviewing（审核中）→ approved（通过）/ rejected（退回）→ completed（完成）
```

- 禁止跳过 reviewing 直接 completed
- AI 生成的产物状态为 ai_generated，**必须经人工审核**方可推进

### 5.3 两级审批流转机制（核心）

每个流程完成（completed）后，向下一流程流转时，须通过**两道审批关**：

```
上阶段 completed → 交付下阶段
  ↓
【第一关·下阶段初审】
  下一阶段接收方，依据上阶段交付的工作成果，人工审是否符合要求
  ├─ 不符合 → 退回上阶段（rejected），上阶段修改后重新提交
  └─ 符合 ↓
【第二关·管理员终审】
  管理员最终审批是否放行
  ├─ 不通过 → 退回
  └─ 通过 → 下阶段正式开始执行（进入下阶段 draft/submitted）
```

**设计要点**：
- **下阶段初审 = "谁用谁把关"**：接收方最懂自己需要什么，先验收上游交付质量，劣质交付物在此被拦下
- **管理员终审 = 统筹放行**：管理员做最终把关和流程调度
- **双重把关**：上阶段不能"自己说完成就完成"，须经下游验收 + 管理员放行

### 5.4 与模板 Workflow Gate 的对应

每个模板 Output 末尾的「是否允许进入下一阶段」准入判断（Workflow Gate），就是本两级审批的落地点：
- **初审** ↔ 下游接收方对该 Gate 的判断（上游交付是否满足本阶段输入要求）
- **终审** ↔ 管理员对该 Gate 的最终确认（放行进入下一阶段）

模板里的「上游依赖声明块」则为初审提供了**验收清单**——接收方据此核对上游是否提供了所需的全部字段。

## 6. 模块执行方式（人/外包/AI 灵活承接）

每个流程可由三种方式之一承接，由**管理员视情况（成本/质量/时效）分派**；无论谁承接，都须经第 5 节两级审批统一把关。

| 承接方式 | 适合的环节 | 怎么做 |
|---------|-----------|--------|
| **AI 承接** | 标准化生成（诊断/方案/分析草稿）| 用模板「AI 生成要点」作提示词生成草稿，**必经人工审核** |
| **外包承接** | 重执行（拍摄/剪辑/设计）| 按模板规范派单，按「审核要点」验收 |
| **人工承接** | 需判断/关系/现场（采集/审批/商家沟通/直播）| 团队成员执行 |

**各流程默认承接建议**（骨架阶段，可调）：

| 流程 | 默认承接方式 |
|------|------------|
| TB-001 诊断 | AI 生成草稿 + 人工审核 |
| TB-002 搭建 | AI 生成草稿 + 运营完善 |
| TB-003 素材采集 | 人工 / 外包（拍摄）|
| TB-004 内容运营 | AI 内容草稿 + 人工/外包制作 |
| TB-005 直播规划 | AI 规划 + 人工执行 |
| TB-006 引流转化 | AI 方案 + 人工执行 |
| TB-007 数据复盘 | AI 分析 + 人工审核（须真实数据）|
| TB-008 90天规划 | AI 生成 + 管理员审核 |

**关键**：承接方式不固定——同一流程，不同商家可用不同方式（A 商家素材外包、B 商家自拍）。这是系统"灵活降本"的核心。

## 7. 多商家并行

- **每个商家 = 一个独立「工单」**，走完整 8 流程，各工单独立、互不干扰
- 系统同时承载多个商家工单，各工单可处于不同流程节点（A 商家在诊断、B 商家在引流）
- **管理员**看到所有工单总览：哪个商家在哪个阶段、卡在哪、待审批什么
- **各角色**看到分派给自己的、跨多个商家的同类任务（如采集员看到所有待采集商家）
- 商家工单**数据隔离**（一个商家的信息/产物不串到另一个）
- **骨架阶段**：先支持少量商家（1-3 个）验证并行管理，跑顺后再扩

## 8. 分成结算设计 ⚠️（按流量分成·初步设定）

> 本节为**初步方向**；分成比例、归因细则、争议处理后续专门细化。
> 但**归因口径是商业命门，须在与商家合作前以书面约定**，否则必生纠纷。

| 项 | 初步设定 |
|----|---------|
| 分成方式 | 按客流量分成 |
| 客流量口径 | ①到店客流（商家提供 POS / 人工统计）②可追踪的线上来源客流（私信咨询、线上优惠核销、线上预约到店、Google Maps 导航点击等）|
| **归因难点（命门）** | 如何区分「代运营带来的客流」与「商家本来的自然客流」|
| 归因初步方案 | ①**基线增量法**：以代运营开始前的基准客流为基线，超出部分作为分成依据；②**可追踪来源法**：只对能明确追踪到线上来源的客流分成——**更清晰、争议少，推荐起步用** |
| 结算周期 | 按月 |
| 初期实现 | **分成结算手工试算**，不入系统（系统初期不做自动结算，呼应"不提前平台化"）|
| 风险控制 | 归因争议 → 合作前书面约定口径，起步优先用「可追踪来源法」降低争议 |

> **✅ 决策（已确认 2026-06-01）：起步采用「可追踪来源法」**——只对"明确是线上运营带来的"客流分成。虽然分成基数小一些，但口径清晰、不扯皮，能让合作先稳稳跑起来；等信任建立、数据完善，再评估是否引入基线增量法。

## 9. MVP 范围（完整骨架优先）

> **本节升级原 MVP_SCOPE**：原定"TB-001 单模板闭环"，现按"完整骨架优先"调整为"完整 8 流程骨架端到端跑通"。

**MVP 目标**：搭出完整 8 流程骨架，让 1 个商家从接入到 90 天规划**端到端走通一次**，验证「流程协作系统」模式成立。

**MVP 包含**：
- ✅ 完整 8 流程节点 + 主流程流转
- ✅ 基础角色登录与权限（采集 / 运营 / 管理员 + AI 生成 + 外包）
- ✅ 两级审批（下阶段初审 + 管理员终审）
- ✅ 状态流转（draft → … → completed）
- ✅ 1 个真实（或模拟）商家工单走完整流程
- ✅ 每个流程"**能跑就行**"的简化版

**MVP 不做**：多租户 / 自动结算 / 自动发布 / 移动 App / 全自动 AI / 大量商家并行 / 每个流程的精细打磨

**MVP 验收标准**：
- [ ] 1 个商家能从接入 → 诊断 → … → 90天规划，完整走通一次
- [ ] 每个流程经**两级审批**才流转到下一环
- [ ] 各角色登录后**只看到自己的任务**
- [ ] AI 能基于模板生成草稿、人工能审核确认/退回
- [ ] 团队能用这套系统**协作干活**（而非靠微信/Excel 手工传递）

## 10. 技术落地路径

**起步：低代码平台搭骨架**

| 项 | 方向 |
|----|------|
| 平台类型 | 低代码/无代码平台（如飞书多维表格+自动化、钉钉宜搭、简道云、明道云等）|
| 能实现 | 分账号登录、任务分派、两级审批、状态流转、工单管理、AI 对接（通过 API/插件）|
| AI 对接 | 把模板「AI 生成要点」做成提示词，通过平台 AI 能力或外部 AI API 调用生成草稿 |
| 为什么低代码起步 | 成本低、速度快（数周可用）、能验证"流程协作"模式，不必先投入开发 |

**升级路径**：模式跑通 + 商家增多 + 需要更复杂功能（如自动结算、对外开放）时，再考虑正经开发独立系统。

**不提前平台化**：不做 SaaS 多租户、不做自动结算、不做对外开放 API（呼应 CODE / SYSTEM_BOUNDARY）。

## 11. 实施路线图

| 阶段 | 内容 | 里程碑/验收 |
|------|------|-----------|
| **一·搭骨架** | 用低代码搭：角色权限 + 8 流程节点 + 两级审批 + 状态流转 | 系统骨架可登录、能流转 |
| **二·跑通** | 用 1 个真实（或模拟）商家，端到端走完整 8 流程，调试流转与审批 | 1 商家完整走通一次（= MVP 验收）|
| **三·服务首个付费商家** | 接入真实付费商家，跑通"收钱→交付→分成（手工试算）"商业闭环 | 第一笔分成收入、商业模式验证 |
| **四·细化** | 按真实使用反馈，逐个流程加深优化（骨架→血肉）| 高频流程优化到位 |
| **五·扩商家** | 支持多商家并行，优化管理员调度与角色协作 | 稳定服务 3-5 个商家 |
| **六（必要时）·独立开发** | 规模化后，考虑正经开发独立系统（含自动结算等）| 视规模与需要决定 |

**节奏原则**：先骨架后血肉、先单商家后多商家、先内部后规模化、先低代码后开发。

## 12. 风险与约束

| 风险 | 说明 | 应对 |
|------|------|------|
| **分成归因争议**（命门）| 客流归因扯皮 | 起步用可追踪来源法 + 合作前书面约定口径 |
| **团队/外包交付质量不齐** | 不同人/外包水平不一 | 两级审批兜底 + 模板标准化（审核要点作验收清单）|
| **AI 产出不可靠** | AI 可能脑补/出错 | 所有 AI 产物必经人工审核，不做全自动闭环 |
| **骨架阶段陷入精雕细琢** | 在骨架阶段抠细节，拖慢跑通 | 守住"能跑就行"，细节留到实施路线图阶段四 |
| **过早做大** | 提前上多租户/自动结算/对外开放 | 不提前平台化，按路线图分步走 |
| **商家不配合** | 商家不提供数据/不配合拍摄 | TB-001 已采集执行限制，分派时考虑；分成依赖数据，需商家配合统计 |

**最高约束**（呼应 CODE）：当前仍是内部工具阶段——先单商家闭环、先低成本验证、不提前规模化。
