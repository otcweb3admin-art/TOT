## ⚠️ 两条最高约束（贯穿全文）

**① Evidence Framework**：所有复盘结论必须有证据支持。**禁止 E0 推测作为复盘结论**——"我觉得有效/可能是因为/大概是这个原因/应该继续做"一律不算。复盘须说明：**依据是什么 · 数据来源 · 执行证据 · 归因依据 · 是否可验证 · 是否可沉淀为经验**。

**② Human Commercial Authority**：AI 可以**整理复盘数据/分析执行结果/识别成功失败因素/提出优化建议/提示经验沉淀机会/提示风险与不确定性**；AI **不可**最终认定成功或失败、自动升级经验、自动改变策略、自动决定续约、自动决定资源投入、自动决定下一轮方向。**最终确认权属于：项目负责人 · 增长负责人 · 公司管理层。**

## 序言：复盘优化是 MGOS 的学习层

> **复盘不是总结报告，不是把数据罗列出来，不是证明自己做得好。复盘是：基于执行证据、监控记录、真实结果、归因判断、目标对比、失败分析，判断——什么有效、什么无效、为什么有效、为什么无效、下一轮如何优化。**

如果没有复盘：成功无法解释 → 失败无法学习 → 经验无法沉淀 → 下一轮优化没有依据 → 增长智能无法形成。因此**复盘与优化是 MGOS 从一次执行进入下一轮增长的学习层**。

## 第一章　Growth Review Definition（增长复盘定义）

### 1.1 复盘不是什么

> 不是总结报告（罗列+自夸）、不是数据堆砌、不是证明做得好。

### 1.2 五概念区分

| 概念 | 回答 |
|------|------|
| **监控 Monitoring** | 过程观察（进行中，承 020）|
| **验证 Validation（MVS）** | 判断结果是否成立（真实 + 可归因）|
| **复盘 Review** | 解释为什么成立/不成立（拆到动作/能力）|
| **优化 Optimization** | 下一轮调整方向 |
| **经验沉淀 Experience** | 把验证过的规律保存（L1→L2+）|

> **复盘是"学习层"**——把"发生了什么 + 是否成立"，变成"为什么 + 下一步怎么更好 + 能沉淀什么经验"。
> **复盘的目的不是汇报，是学习**：诚实解释成败（承 MVS 诚实）。**证明自己做得好 = 自欺 = 污染。**

---

## 第二章　Monitoring-to-Review Bridge（监控到复盘的桥）

复盘必须承接监控（TASK-020）的输出：当前执行状态 · 数据表现 · 正向信号 · 异常信号 · 偏差记录 · 风险提示 · 数据缺口 · 纠偏记录 · 是否影响 MVS 验证。

| 监控输出 | 复盘如何使用 |
|---------|-------------|
| 信号/异常 | → 复盘的"待解释清单" |
| 偏差/纠偏记录 | → 判断"执行是否到位" |
| 是否影响 MVS | → 判断"归因是否可信" |

> **没有监控记录，复盘容易事后编故事。** 监控的过程记录是复盘"不编故事"的保障——有过程证据，复盘才是"**还原真相**"而非"事后叙事"。没监控记录，人会用结果倒推一个好听的故事（**后见之明偏差**）。

---

## 第三章　Review Input Model（复盘输入模型）

复盘是**全链输入** + 纳入外部以排除外因。至少依赖：

```
诊断结果 · 已确认策略 · 增长计划 · 执行记录 · 执行证据 · 监控记录 · KPI 目标
· 真实经营结果 · 归因记录 · 偏差记录 · 商家反馈 · 客户反馈 · 市场变化 · 竞品变化 · 历史经验
```

> - 复盘要对照**整条增长链**（诊断/策略/计划/执行/监控）看哪一环出问题（承 Ch4）。
> - 纳入市场/竞品变化以**排除外因**（承 MVS 混淆排除）。
> - **缺关键输入时，不得输出确定性复盘结论。**

---

## 第四章　Review Object Model（复盘对象模型）

> **复盘不能只复盘总结果。复盘必须覆盖整条增长链，而不是只看最终数字。**

| 复盘对象 | 问什么 | 不达标指向 |
|---------|--------|-----------|
| 诊断 | 是否准确（找对问题了吗）| 重新诊断 |
| 策略 | 是否正确（打对仗了吗）| 改向 |
| 计划 | 是否合理（路径对吗）| 调计划 |
| 执行 | 是否到位（做到了吗）| 纠偏 |
| 监控 | 是否及时（早发现了吗）| 优化监控 |
| 归因 | 是否清晰（证明得了吗）| 补钩子/数据 |
| KPI | 是否达成 | 看根因 |
| 商家配合 | 是否充分 | 沟通/约定 |
| 内容/渠道/Campaign | 是否有效 | 优化对应环节 |
| 风险 | 是否被提前识别 | 优化风控 |

> **只看"涨没涨"是浅复盘**；深复盘逐环问"诊断对吗？策略对吗？……执行到位吗？"。**同样的"没增长"，可能是诊断错（找错问题）/策略错（打错仗）/计划错（路径错）/执行没到位（没做）——四种根因要不同优化。** 全链复盘 = 定位"链条哪一环断了"，是"**对整个周期的诊断**"。

---

## 第五章　Success Analysis Framework（成功分析框架）

出现增长结果时，必须回答：

```
增长是否真实 · 是否可归因 · 来自哪个动作 · 来自哪个渠道 · 是否可复现 · 是否可扩大 · 是否有成本代价 · 是否可沉淀为经验
```

> **成功不等于经验。成功必须经过 MVS 验证，才能进入 Experience Base。**
> - 承接 MVS：增长真实 + 可归因（MVS 判定）→ 复盘进一步拆"来自哪个动作/渠道"（归因细化）→ 判断可复现/可扩大（经验候选潜力）。
> - **区分"真成功"（真实 + 可归因 + 可复现）vs"假成功"（涨了但不可归因 / 一次性运气 / 季节）**——只有真成功才有经验价值。
> - 产出：成功的"**条件→动作→结果**"三元组（承 Experience Base 案例结构），作为经验候选。

---

## 第六章　Failure Analysis Framework（失败分析框架）

失败必须分类（10 类）：

| 失败类型 | 优化方向 |
|---------|---------|
| 诊断失败 | 重新诊断（根本性）|
| 策略失败 | 改向（根本性）|
| 计划失败 | 调计划 |
| 执行失败 | 纠偏（方向可能对）|
| 监控失败 | 优化监控 |
| **归因失败** | 补钩子/补数据（**最该避免**）|
| 商家配合失败 | 沟通/约定 |
| 市场外部因素 | 记录但不否定方案 |
| 资源不足 | 资源决策（人）|
| 时间不足 | 周期调整 |

> **失败不是垃圾数据。失败是 Experience Base 中非常重要的负经验来源**（承 Experience Base"失败模式同等准入"）。失败必须被记录、解释、归类。
> - **失败归类决定优化方向**：诊断/策略失败→根本性（重诊/改向）；计划/执行/监控失败→非根本（调整）；外部/资源/时间→非方案问题（记录不否定方案）。
> - **最该避免的是"归因失败"**（增长发生了但不知道是不是自己的/为什么）——它让这个周期"**学不到东西**"（承 MVS"不知道为什么失败"是最无价值的）。
> - 产出：失败的"**条件→动作→结果(失败)→原因**"作为负经验候选。

## 第七章　Optimization Framework（优化框架）

| 优化动作 | 适用条件 | 是否影响归因 | 谁确认 |
|---------|---------|:-----------:|--------|
| **继续** | 有效且可复现 | 否 | 可自行 |
| **微调** | 基本有效，小改不动关键变量 | 否（须确认）| 谨慎 |
| **纠偏** | 执行没到位 → 拉回计划 | 否（保护）| 可自行 |
| **放大** | 验证有效 → 扩大规模/预算 | **是（新变量）** | 须人 + 重设基线 |
| **暂停** | 效果不明/风险高 | 否 | 须人 |
| **放弃** | 验证无效且非执行问题 | 切断 | 须人 |
| **重新诊断** | 诊断错了 → 回诊断层 | 重置 | 须人 |
| **重新制定策略** | 策略错了 → 回策略层 | 重置 | 须人 |

> - **优化不是随便改。优化必须保护 MVS 的验证有效性。**
> - **优化对应复盘归类**：成功且可复现→放大；执行失败→纠偏；策略失败→改向。优化是"基于复盘归类的对症处置"。
> - **放大/改向是新变量，须重设验证基线**（承 monitoring/planning）；不为短期效果乱改变量。

## 第八章　Review Quality Gate（复盘质量门）

一份合格复盘**必须满足**：

```
☑ 有目标对比   ☑ 有执行证据   ☑ 有监控记录   ☑ 有结果数据   ☑ 有归因判断
☑ 有成功/失败解释   ☑ 有偏差分析   ☑ 有优化建议   ☑ 有经验沉淀判断   ☑ 有人工确认点
```

> 不满足 → **不得进入经验沉淀阶段**。
> 这是学习层的质量门，**防"事后编故事的复盘"**——没有执行证据/监控记录支撑的复盘结论 = E0 叙事，不得沉淀。

---

## 第九章　Experience Admission From Review（复盘到经验的准入）

引用 Evidence Framework / MVS / Experience Base V2 / Experience Application。**只有满足以下条件，才允许进入经验候选**：

```
有真实执行 · 有执行证据 · 有结果数据 · 有归因判断 · 有上下文 · 有边界条件 · 有失败风险说明 · 经人工确认
```

> **复盘结论不是经验，复盘结论只是经验候选。**
> - 复盘产出的"条件→动作→结果"三元组 = 填充 Experience Base **八要素**（Context/Action/Outcome/Evidence/Confidence/Boundary/Failure Risk，承 v2）。
> - 复盘是经验库的"**入口生产线**"：把一个周期的成败炼成"经验候选（L1 案例）"，但**升级为经验（L2+）要经 Experience Base 准入门槛**（多案例 + 同场景 + 一致 + 可复现 + 人审定）。**复盘不能自己宣布"这是经验"。**
> - **防污染**：不可归因的成功 / 未验证的成功，不得进经验候选（承 MVS / Experience Base）。

## 第十章　Next Cycle Strategy Input（下一轮输入）

复盘输出应成为下一轮的：

```
诊断修正 · 策略调整 · 计划优化 · 执行标准优化 · 监控指标优化 · 经验召回依据 · 商家成功管理依据
```

形成闭环：

```
Review ──► Optimization ──► Next Diagnosis / Strategy / Plan ──► ...（新一轮）
```

> 这是 MGOS"**周期性反馈控制**"的闭合点（承 MGOS Ch3）——**一轮的复盘是下一轮的输入，让每轮比上轮更准**（承 growth-intelligence 飞轮 / TB-008→循环回 TB-007）。

## 第十一章　Human Review Principle（人工复盘确认原则）

> **复盘与优化必须经过人工确认。**

| AI 负责 | 人负责 |
|---------|--------|
| 整理复盘材料、识别成功/失败因素、识别偏差、提出优化建议、标记经验候选 | **确认复盘结论、判断是否进经验候选、判断下一轮是否调策略、是否增资源、是否续约或终止、承担最终责任** |

- **AI 不可**：最终认定成败、自动升级经验、自动改策略、自动决定续约/资源/下一轮方向。
- **为什么复盘要人确认**：复盘结论决定续约/资源/方向/经验——重大商业判断；且**复盘易有自利偏差**（想证明自己做得好），人确认是诚实的兜底。与 AIGO + Human Commercial Authority 一致。

### 12.1 复盘如何避免经验污染

> 质量门 + 只有"验证过 + 可归因 + 人审定"才进候选；**失败诚实记录防幸存者偏差**。
> **核心警示**：没有复盘 → 成功无法解释 / 失败无法学习 / 经验无法沉淀 / 下一轮无依据 / 增长智能无法形成。

## 第十三章　Architecture Conclusions（架构结论）

1. **复盘是学习层，不是总结报告**：诚实解释成败 + 优化 + 产经验候选。
2. **复盘必须全链覆盖**：逐环复盘（诊断/策略/计划/执行/监控/归因），不只看最终数字。
3. **成功 ≠ 经验**：成功须经 MVS 验证、可归因、可复现，才有经验价值；区分真成功与假成功。
4. **失败同等重要**：10 类失败归类，失败模式是高价值负经验；归因失败最该避免（学不到）。
5. **优化是对症处置**：八种动作对应复盘归类；放大/改向是新变量，须重设基线。
6. **优化保护归因**：不为短期效果乱改变量；改向重新确认验证条件。
7. **复盘结论 ≠ 经验，只是经验候选**：填八要素入 L1，升 L2+ 须经 Experience Base 准入门槛 + 人审定。
8. **复盘质量门**（十项）防"事后编故事"，不达标不进经验沉淀。
9. **复盘闭合循环**：Review→Optimization→下一轮诊断/策略/计划，是飞轮的"学习"环节。
10. **AI 整理、人确认**：AI 不认定成败/不自动升级经验/不决定续约·资源·方向。

### 13.1 六个根本问题的回答

| 问题 | 回答 |
|------|------|
| **为什么复盘不是总结报告？** | 总结是罗列+自夸；复盘是诚实解释成败 + 学习 + 优化。|
| **为什么复盘必须解释成功和失败？** | 不解释为什么 = 无法复现成功、无法避免失败 = 学不到。|
| **为什么失败经验同样重要？** | 失败模式（"X 条件做 Y 会失败"）是高价值负经验，防后续踩坑（防幸存者偏差）。|
| **为什么复盘结论不能直接成为经验？** | 复盘结论是单周期/单案例 = L1 候选；经验（L2+）要多案例同场景一致可复现 + 人审定。|
| **为什么复盘必须保护 MVS 归因？** | 复盘基于 MVS 归因；优化（放大/改向）乱改变量 → 下轮归因失真。|
| **为什么没有复盘，系统不会成长？** | 不复盘 = 不学习 = 每轮从零 = 无飞轮 = 停在"案例公司"。复盘是飞轮转动的那一下"学习"。|

### 13.2 最终结论

> **复盘与优化是 MGOS 从一次执行进入下一轮增长的学习层。** 复盘 = **诚实解释成败 + 优化方向 + 产经验候选（而非经验）**——它**全链复盘、成功与失败同样分析、保护归因、人确认**。**没有复盘，增长智能无从形成**——它是飞轮转动的那一下"学习"，是 TOT 从"案例公司"走向"增长智能公司"的引擎。复盘合上了增长运行的循环：一轮的学习，是下一轮的起点。

```
诊断 ──► 策略 ──► 计划 ──► 执行 ──► 监控 ──► 【本文：复盘优化（学习层）】──► 验证(MVS) ──► 经验候选
门1      门2      门3      门4    (过程反馈)   全链复盘·成败分析·优化·产候选        │
 ▲                                                                              │ 经人审定升级
```
