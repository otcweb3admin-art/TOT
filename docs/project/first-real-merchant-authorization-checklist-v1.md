# First Real Merchant Authorization Checklist V1（首家真实商家·接入授权检查清单）

> 文档类型：纯业务 / 风控 / 操作授权清单（TASK-077）。不改代码、不建真实商家、不建真实账号、不写真实客户信息。
> 配套文档：[试运行 Runbook](./first-real-merchant-pilot-runbook-v1.md)（授权之后怎么做）· 本文（**是否允许开始**）。
> 最高约束：AI 不拍板（Human Commercial Authority）——本清单由**人**逐项勾选，由**人**签发结论。

---

## Chapter 1：Purpose（目的）

这是首家真实商家进入 TOT 系统前的**最终授权清单**。它不是销售文案，不是试点计划，而是"**是否允许把真实商家带入系统**"的放行门。

它决定五件事：是否允许创建真实商家 / 是否允许创建客户账户 / 是否允许采集真实资料 / 是否允许进入首轮试运行 / 哪些条件不满足时必须暂停。

核心判断（五句话，逐级闸门）：

```text
没有授权，不建真实商家。
没有账号准备，不进入系统。
没有资料边界，不采集。
没有证据纪律，不诊断。
没有人工审核，不进入下一步。
```

从此**不能因为"想试试"就录入真实商家**——必须先过本清单。

## Chapter 2：Authorization Decision Owner（谁有权放行）

| 主体 | 在授权中的位置 |
|------|--------------|
| 系统 | **不自动决定**是否接入真实商家（系统就绪 ≠ 自动开始） |
| AI / ai_worker | **不参与授权决策**（只出草稿，永不拍板） |
| admin 平台管理账号 | **不是自动授权人**——有 admin 权限 ≠ 有商业放行权 |
| 外包 executor | 不参与授权决策 |
| 采集员 collector | **不能自行决定接入**（即便系统权限允许其创建商家，授权前不得创建真实商家） |
| 审核员 operator | **可以提出建议**（就绪评估 / 风险提示），不拥有最终放行权 |
| **商业负责人 / 项目负责人（人）** | **唯一最终放行人** |

**结论：**

```text
真实商家接入必须由人工商业负责人明确授权。
```

授权必须落到 Chapter 11 的授权记录上（有名字、有日期、有范围），口头"可以吧"不算授权。

## Chapter 4：Internal Account Readiness Checklist（内部账号准备）

按 [内部账号指南](./internal-role-account-setup-guide-v1.md) 人工准备；密码线下交付，**绝不写入本清单**。

**admin（平台管理）**
- [ ] 是否有真实平台管理账号（非临时夹具）
- [ ] 是否避免多人共用（一人一号）
- [ ] 是否明确**不作为日常运营账号**

**operator（人工审核）**
- [ ] 是否有人工审核账号且知道工作台入口
- [ ] 是否明确其负责 **AI 草稿审核**
- [ ] 是否明确其负责**外包验收**
- [ ] 是否明确其负责**客户确认处理**（发起 / 看反馈 / 收口）

**collector（采集员）**
- [ ] 是否有采集员账号
- [ ] 是否知道 **Field Pack**（线下采集→系统录入顺序）
- [ ] 是否知道**证据标注规则**（来源 / 口述 / 估算 / 待补充）

**executor（外包 / 执行）**
- [ ] 是否有外包账号
- [ ] 是否知道**只看得到自己的任务**
- [ ] 是否知道按**验收标准**提交
- [ ] 是否知道**不得接触完整经营数据**、不直接联系客户（除非授权）

**merchant（客户）**
- [ ] 是否需要给客户创建账号（首轮可后置到发起首个确认事项前，但须在此明确）
- [ ] 客户联系人是否明确（与 Ch3 #10 一致）
- [ ] 客户是否知道登录后**只看自己的事项**

**ai_worker**
- [ ] 是否明确**不作为真人账号**（无人持有其登录方式）

## Chapter 5：Data Boundary Checklist（数据边界）

- [ ] 是否确认**哪些资料可以录入系统**（授权范围内、服务所需最小集合）
- [ ] 是否确认**哪些资料不能录入系统**（授权范围外的隐私 / 财务明细 / 与服务无关信息）
- [ ] 是否确认**敏感信息处理方式**（谁可见、存在哪个字段、是否需要脱敏 / 摘要化）
- [ ] 是否明确**口述信息标「口述」**
- [ ] 是否明确**估算信息标「估算」**
- [ ] 是否明确**不确定信息标「待验证 / 待补充」**
- [ ] 是否明确 **AI 输出不能当事实**（E0 模型推测，必须人工核实或标待验证）
- [ ] 是否明确 **UAT / DEMO 不可作为真实案例**（不出现在任何对客户/对外材料）
- [ ] 是否明确**不得录入未授权隐私**（发现即停、上报负责人处理）

## Chapter 6：Pilot Scope Checklist（首轮范围）

- [ ] 首轮只做**小范围闭环**（资料 → 诊断 → 一批内容/素材 → 客户确认）
- [ ] 首轮**不追求完整增长结果**（目标=流程跑通+资料准确）
- [ ] 首轮优先验证：资料采集 / AI 草稿 / 外包 / 客户确认 四条流程
- [ ] 首轮**不盲目投流**（投流须另行人工决策）
- [ ] 首轮**不承诺 ROI**
- [ ] 首轮**不承诺客流**
- [ ] 首轮**不承诺销售额**
- [ ] 首轮**不做大规模内容发布**（发布须客户确认 + 负责人同意）
- [ ] 首轮结束**必须复盘**（按 Runbook Ch15 完成标准收口）

## Chapter 7：System Readiness Checklist（系统就绪）

> 以仓库与最近验证为准（UAT 彩排报告 / 生产 RC 验收）；逐项人工核对当下状态，不凭印象。

- [ ] UAT 全链彩排是否 **PASS**（参考：2026-06-12 报告 51/51 PASS）
- [ ] 任务中心是否可用（创建 / 分配 / 提交 / 退回 / 通过 / 完成）
- [ ] AI 草稿审核是否可用（工作台提交 → 待审核 → 人审 → 人工保存入口）
- [ ] 外包提交是否可用（成果说明 / 链接 / 退回重交）
- [ ] 客户确认是否可用（发起 → 确认 / 提意见 → 反馈回审核员）
- [ ] 未登录访问是否仍跳转 login（守卫生效）
- [ ] 六类角色首页是否正常（各自工作台与导航）
- [ ] 当前测试数据是否清理干净（`npm run uat:check-clean` = CLEAN;无 SMOKE 残留）
- [ ] DEMO / UAT / 真实数据边界是否清晰（前缀与徽章可辨）
- [ ] 是否存在未解决阻断问题（有 → 列出并先解决或降级处理）

## Chapter 8：First Day Go / No-Go Checklist（首日放行）

**Go 条件（须全部满足）**：
- [ ] 商家授权已确认（Ch3 全过）
- [ ] 内部负责人已确认（有名字）
- [ ] 账号已准备（Ch4 全过）
- [ ] 采集员已准备（知道 Field Pack 与证据规则）
- [ ] 审核员已准备（知道审什么、怎么退回）
- [ ] 数据边界已确认（Ch5 全过）
- [ ] 首日目标明确（资料准确 + 流程启动，仅此而已）
- [ ] 全员不承诺增长
- [ ] 不急于外包
- [ ] 不急于客户确认
- [ ] 当天只做资料准确性和流程启动

**No-Go 条件（任一出现即停）**：
- [ ] 商家未授权
- [ ] 商家期望被承诺增长（先按 Outreach Kit 重新对齐预期，否则不进）
- [ ] 内部账号未准备
- [ ] 没有人负责审核
- [ ] 不知道谁联系客户
- [ ] 数据来源不清
- [ ] 准备直接投流
- [ ] 准备把 AI 输出直接给客户
- [ ] 准备用真实客户做系统调试（调试用 UAT_/DEMO_，永远如此）

**Go 条件**：

## Chapter 12：Final Decision States（最终决策状态）

```text
GO
CONDITIONAL GO
NO-GO
```

- **GO**：所有关键条件满足（Ch3~Ch8 全过）→ 可以创建真实商家并进入首日流程（随后严格按 Runbook 执行）。
- **CONDITIONAL GO**：允许进入**非常小范围**流程，但必须在授权记录里写明缺失项与补齐期限。典型情形：
  - 账号部分未齐（如 executor 未到位 → 本轮外包暂不参与）；
  - 客户账号暂缓 → **客户确认环节暂不开放**，到发起首个确认事项前必须补齐；
  - 外包暂不参与 → 内容先内部产出；
  - 部分资料待补充 → 可先采集已授权部分，缺口标注待补充。
  CONDITIONAL GO 的范围之外**一律视为未授权**。
- **NO-GO**：**不得创建真实商家。** 典型情形：未授权 / 商家要求承诺增长 / 内部无审核员 / 数据边界不清 / 准备拿真实客户调试系统。NO-GO 不是失败——回到 Outreach Kit / Prospect Tracker 继续推进，条件齐了再来过本清单。

## Chapter 13：Final Recommendation（最终建议）

**当前系统已具备首家真实商家试运行能力（UAT 彩排 PASS + Runbook 就绪），但真实商家进入必须经过本授权清单。**

建议执行顺序：
1. 选择候选商家（Prospect Tracker 中兴趣度高、画像匹配者）
2. 完成授权确认（Outreach Kit 话术对齐预期 → Ch3 逐条确认）
3. 填写本 Checklist（Ch3~Ch10 逐章勾选）
4. 得到结论：GO / CONDITIONAL GO / NO-GO（Ch11 模板留档）
5. **只有 GO，或明确限定范围的 CONDITIONAL GO，才允许进系统**
6. 进系统后严格按 [Runbook](./first-real-merchant-pilot-runbook-v1.md) 执行（25 步顺序 / 首日清单 / 首周清单 / 风险处理 / 完成标准）
