## 第一章　Why Experience Must Be Applied（为什么经验必须被应用）

### 1.1 一个不被调用的经验库，是死资本

Experience Base v1/v2 建好了库——经验如何形成、验证、升级、治理。但**库若从不被调用，再多经验也只是"博物馆藏品"**：看着丰富，对增长毫无作用。

> **经验的价值不在"存了多少"，而在"被用了多少、用得多准"。** 应用层是经验库从"静态资产"变成"活的增长大脑"的那道工序。

### 1.2 应用层填补的四个空白

| Experience Base 已解决 | 应用层（本文）要解决 |
|----------------------|---------------------|
| 经验如何形成/验证/升级/治理 | 经验如何**被召回** |
| | 经验如何**影响方案** |
| | 经验如何**参与审计** |
| | 经验如何**参与增长决策与风控** |

### 1.3 应用层是"经验公司→增长智能公司"的兑现点

experience-base-v2 指出：经验库（存）≠ 增长智能（用）。**本文就是那个"用"**——它把"1000 家 > 10 家"从理论变成现实：新商家因为前人经验被**召回、排序、适配、注入方案**而开局即有据。**没有应用层，飞轮的"消费端"是断的。**

## 第二章　Experience Recall Architecture（经验召回架构）

### 2.1 触发点：TB-001 完成后

当 TB-001 完成，一个商家就有了完整的 **Context 向量**（来自 Merchant Profile Asset）：

```
商家 Context = 行业 × 城市/国家 × 客单价 × 渠道成熟度 × 增长目标 × 增长阶段 × 用户群体 …
```

**召回 = 用商家 Context 匹配经验库中"同切片"的经验**（接 experience-base-v2 第七章切片维度）。

### 2.2 分层召回机制（Context → Experience Recall）

经验库里可能有海量经验，召回**按匹配精度分层递进**：

```
① 精确匹配：同行业 × 同城 × 同客单 × 同目标 …  → 最贴近的 L2 同场景经验（最高优先）
② 放宽匹配：同行业 × 同客单（放宽地理/阶段）    → L2/L3 经验
③ 上层匹配：同行业                              → L3 行业经验
④ 跨层匹配：跨行业可迁移                         → L4 跨行业模式
⑤ Fallback：无任何匹配                          → 诚实标"无经验，靠方法论起步"
```

### 2.3 召回四原则（业务层，非检索算法）

1. **上下文优先**：切片匹配是召回的基础——**先看"对谁适用"（Context/Boundary 八要素），再看内容**。
2. **分层递进**：精确不足则放宽，放宽不足则升上层，最后回落方法论。
3. **宁缺毋滥**：无匹配时**诚实标"无经验"**，绝不硬塞不相关经验（防误迁移，呼应 v2 第七章）。
4. **成功与失败同召回**：失败模式经验同等召回——它是第七章风险预警的来源。

> **输出**：一个**候选经验集**，每条标注匹配层级（精确/放宽/上层/跨层）+ 其八要素。**召回只负责"找出可能相关的"，不负责"决定用哪条"**——那是排序与人工的事。

## 第三章　Experience Ranking Architecture（经验排序架构）

> 召回后可能有 10 / 100 / 1000 条。需要排序，让"最相关 + 最可信"的浮到最前。**仅给业务优先级原则，禁止设计算法/权重公式。**

### 3.1 Experience Priority Model（优先级思想模型）

优先级由**两组维度共同决定**（定性分层，非数字公式）：

| 维度组 | 维度 | 推高优先级 |
|--------|------|-----------|
| **匹配度** | 行业匹配 / 城市匹配 / 阶段匹配 | 越精确匹配越优先 |
| **可信度** | 证据强度(E3>E1>E2) / 成功率 / 一致性 / 验证次数 / 样本量 / 时间新鲜度 / 边界清晰度 | 越硬越优先（接证据可信度模型 + v2 质量框架）|

### 3.2 排序的定性分层（非公式）

```
高优先：精确匹配 + Strong Experience（多样/一致/新近/可复现/边界清晰）
中优先：放宽匹配 + Medium，或 精确匹配但证据中等
低优先：上层/跨层匹配，或 Weak Experience（仅作提示，不主导）
过期：时效失效经验 → 降到最低 / 标"待重验"
```

### 3.3 排序三原则

1. **匹配度与可信度都要看**：高匹配但低置信，未必胜过中匹配但强证据——**两者并呈，交人工/Planner 判断**。
2. **全部带标签浮现**：每条经验**始终携带匹配层级 + 置信 + 时效**，排序只是"建议顺序"，不隐藏信息。
3. **质量 ≠ 数量**：召回到 100 条同源弱经验，不如 3 条多样强经验靠前（接 v2 第八章）。

> **禁止**：把排序做成黑箱评分。排序是**给人/Planner 的"建议优先级"**，必须可解释、可追溯到"为什么这条排前面"。

## 第四章　Conflict Resolution Framework（冲突化解框架）

### 4.1 核心洞察：经验冲突 ≠ 经验错误

示例：
- 经验 A：直播有效
- 经验 B：直播无效
- 经验 C：仅部分场景有效

> **多数"冲突"不是经验错了，而是切片不够细——A 和 B 其实属于不同场景。**（接 experience-base-v1 冲突治理）

### 4.2 冲突化解四步

```
① 先查切片：A、B 是否同场景？
     ├─ 不同场景 → 不是冲突，是【场景分化】→ 各自适用各自场景，补细切片维度
     └─ 同场景   → 进入②（真冲突）
② 证据仲裁：按 证据强度 / 样本量 / 验证次数，高置信者优先
③ 时效仲裁：新近经验 > 过期经验（如平台规则已变）
④ 无法仲裁：标"不确定"，交人工判断，【不强行二选一】
```

### 4.3 冲突处理原则

| 原则 | 含义 |
|------|------|
| **冲突是信号，不是 bug** | 冲突往往提示"该细分场景"或"该重验某条" |
| **不删除，标边界** | 输出不是"删掉一条"，而是标注各自适用边界 + 置信 + 冲突原因 |
| **场景分化优先于二选一** | 先问"是不是不同场景"，再谈"谁对谁错" |
| **真冲突交人工** | 无法以证据/时效仲裁时，呈现给人，不 AI 拍板 |

> **冲突是经验库变精细的契机**：经验 C"仅部分场景有效"恰恰是 A、B 冲突被正确化解后的产物——**化解冲突 = 加细切片维度**。

## 第五章　Experience-Driven Planning（经验驱动的方案生成）

### 5.1 完整链路

```
Merchant Profile ──► Experience Recall ──► Experience Ranking ──► Experience Adaptation ──► Strategy Draft ──► Human Review ──► Final Plan
（TB-001 画像）      （第二章）            （第三章）             （适配，关键）         （Planner草稿）    （含经验审计）    （人批准）
```

| 环节 | 做什么 | 谁（AIGO）|
|------|--------|----------|
| Merchant Profile | TB-001 画像 + 切片 | Growth Analyst AI |
| Recall | 召回同场景候选经验 | Growth Analyst AI |
| Ranking | 按匹配度+可信度排序 | Growth Analyst AI |
| **Adaptation 适配** | **把经验裁剪到本商家约束**（红线/预算/执行能力/资源）| Strategy/Content/Campaign Planner AI |
| Strategy Draft | 生成方案草稿（每条标"依据哪条经验+置信"）| Planner AI |
| Human Review | 审核（含经验审计，第六章）| 运营/管理员 |
| Final Plan | 批准后的方案 | Human |

### 5.2 Adaptation 是关键：经验是蓝本，不是复制粘贴

> 即使召回到"同场景成功经验"，也**不能照搬**——必须按本商家的 **Boundary（红线/预算/执行能力）裁剪 + 降级**。

- 经验说"做高频探店直播有效"，但本商家红线"老板不出镜" → 适配为"员工/产品视角的探店"，或降级方案。
- **经验提供"该往哪走"，适配解决"这家店能怎么走"**——脱离本商家约束的经验照搬，正是 AIGO 禁止的"能力幻想"。

### 5.3 与 MGOS 的对接

这条链路就是 **MGOS 五段闭环的 Diagnosis→Strategy 段**的展开——**经验在此被注入方案**。但产物仍是 `ai_generated` 草稿，**经人工审批才成 Final Plan**（不 AI 拍板）。

## 第六章　Experience-Aware Audit（经验感知审计）

### 6.1 审计的升级：从形式合规到实质有效

AIGO 的 Auditor（Risk/Evidence Auditor AI）原本检查**模板完整性 + 证据充分性**。经验应用后，**还必须检查：方案是否违背高可信历史经验**。

### 6.2 经验感知审计三查

| 查项 | 触发 | 处置 |
|------|------|------|
| **① 是否撞上高置信失败经验** | 方案在做"已验证会失败"的事 | **强预警 / 拦截建议**（附失败原因 + 替代）|
| **② 是否忽略高置信成功经验** | 方案没用"已验证有效"的打法 | 提示（非强制，可能有合理理由）|
| **③ 是否超出经验适用边界** | 把某场景经验误用到不匹配场景 | 预警误迁移（呼应第四章）|

### 6.3 原则

- **经验审计是"建议"，不是"否决"**：Auditor 标记"方案与经验不符"，但**最终仍由人裁定**（方案可能有经验未覆盖的合理创新）。
- **失败经验在此兑现价值**：审计撞上失败经验 → 提前拦下"明知会失败"的方案。
- **审计可追溯**：每条经验符合性判断，可追溯到具体经验 + 其置信。

> **Experience-Aware Audit 让审计从"格式对不对"升级到"做法历史上灵不灵"**——这是审计的实质化。

## 第七章　Experience-Based Risk Prevention（基于经验的风险预防）

### 7.1 示例：某方案历史失败率极高 → 提前预警

风险预防的核心：**让"踩过的坑"被提前绕开**。

```
方案动作 ──匹配──► 高失败率历史经验 ──► 提前预警 + 失败原因 + 降级/替代建议
```

### 7.2 经验风险预警分级

| 匹配到的失败经验 | 预警级别 | 处置 |
|-----------------|---------|------|
| 高置信失败经验（Strong）| **强预警** | 建议不做 / 必须改 |
| 中置信失败经验（Medium）| 提示 | 注意，须佐证 |
| 弱置信失败经验（Weak）| 记录 | 留痕，供参考 |

### 7.3 与 MGOS 七层防御的关系

MGOS 七层防御（红线/铁律/证据/审批/降级/验证/治理）是**规则防御**；经验风险预防是其上的**"历史防御层"**：

> 前七层防"违反规则"，经验层防"重蹈覆辙"——**规则告诉你"不能做什么"，经验告诉你"这样做过、结果很差"**。两者互补。

> **预防是"提前"，不是"事后"**：经验让系统在方案阶段就预警高风险动作，而非等执行失败后复盘。**失败模式经验的最大价值，在这里兑现。**

## 第八章　Continuous Learning Loop（持续学习闭环）

### 8.1 闭环：应用 → 结果 → 验证 → 更新 → 更好的应用

```
应用经验(本文) ──► 执行 ──► Result(真实数据) ──► Validation(MVS) ──► Experience Update ──► Future Decision
                                                       │                    │                    │
                                                  归因(执行/策略)      入库/升级/降权        下一商家召回更准
```

| 环节 | 谁负责 | 一致于 |
|------|--------|--------|
| Result | 真实经营数据(E1) | MVS |
| Validation | 归因（执行 vs 策略）| **MVS（TASK-007）** |
| Experience Update | 强化/修正/新增/降权 | **Experience Base（v1/v2）** |
| Future Decision | 召回更准、智能更强 | **Growth Intelligence** |

### 8.2 关键：应用即贡献（经验的自我修正）

- **每次应用都产生可回流的新案例**：经验被用 → 产生新结果 → 验证 → 回流强化或**修正**原经验。
- **经验被自己的应用结果检验**：若应用某经验后结果不及预期 → **该经验置信下调**。这让经验库**自我纠错**（self-correcting）——用得越多，不仅越准，还越"诚实"。
- **应用不是单向消费，是双向闭环**：消费端（召回应用）与生产端（验证沉淀）首尾相连，正是飞轮转动（呼应 growth-intelligence-v1）。

> **闭环保持三系统一致**：MVS 验证、Experience Base 更新、Growth Intelligence 供能——本文是这三者在"应用侧"的接口。

## 第九章　Application Boundary（经验应用边界）

### 9.1 经验不能代替判断

> **经验只能：提供参考 · 提供概率 · 提供风险提示。**
> **经验不能：自动拍板 · 自动批准 · 自动决策。**

### 9.2 边界五原则

| 原则 | 含义 |
|------|------|
| **经验是参考，不是指令** | 召回排序的经验供人/Planner 参考，不自动执行 |
| **经验提供概率，不提供确定** | "此场景这样做 70% 成功" ≠ "一定成功"；仍需人判断是否适用本商家 |
| **经验有适用边界** | 超边界的经验不适用（第四/六章）|
| **经验不拍板** | 最终方案/审批/决策恒在人（AIGO Authority Model）|
| **无经验时诚实** | 没有匹配经验就靠方法论起步，**绝不编造经验** |

### 9.3 最大的风险是"经验崇拜"

> 经验应用的最大陷阱，是把历史经验当圣旨——**"以前这样成功过，所以这次也一定"**。这违背证据宪法（经验是 E3，仍须结合本商家 E1）、违背增长智能（经验有边界与时效）。

**Boundary 就是防经验崇拜**：

> **经验是地图，不是方向盘。** 它告诉你"前人走过的路通向哪里"，但**这辆车怎么开、要不要走这条路，由人决定。** 这与 Evidence Framework（经验只是一种证据，不替代决策）和 AIGO（AI 不拍板）完全一致。

## 第十章　Architecture Conclusions（架构结论）

1. **不被应用的经验库是死资本**：应用层是经验从"静态资产"变"活的增长大脑"的工序，是"经验公司→增长智能公司"的兑现点。
2. **召回以 Context 为基础、分层递进**：精确→放宽→上层→跨层→方法论 fallback；宁缺毋滥、成功失败同召回。
3. **排序看匹配度 + 可信度，仅给业务原则不给算法**：可解释、可追溯、全部带标签浮现；质量 ≠ 数量。
4. **冲突 ≠ 错误**：多数冲突是切片不够细；先场景分化、再证据/时效仲裁、无解交人工；冲突是经验变精细的契机。
5. **经验驱动方案的关键是 Adaptation**：经验是蓝本不是复制粘贴，必须按本商家红线/预算/能力裁剪降级。
6. **审计升级为 Experience-Aware**：除模板/证据，还查"是否违背高可信经验"，但是建议非否决，最终人裁定。
7. **风险预防加"历史防御层"**：匹配高失败率经验则提前预警；规则防"违规"，经验防"重蹈覆辙"。
8. **持续学习闭环**：应用→结果→MVS 验证→经验更新→未来更准；应用即贡献，经验自我修正。
9. **经验有应用边界**：提供参考/概率/风险提示，不自动拍板；防"经验崇拜"——经验是地图不是方向盘。
10. **全程与证据宪法/AIGO 一致**：经验是一种 E3 证据，只支撑决策不替代决策，AI 不拍板，人决策。
11. **应用层是飞轮的消费端**：让"1000 家 > 10 家"从理论变成新商家"开局即有据"的现实。

### 10.1 与 Experience Base V2 的关系

| Experience Base V2（库）| Experience Application V1（用）|
|------------------------|------------------------------|
| 经验如何形成/结构(八要素)/层级(L0–L5)/升级/治理 | 经验如何召回/排序/化冲突/进方案/进审计/进风控/学习 |
| **静态：库存在哪、对不对** | **动态：库如何参与每一次增长决策** |

> 关系：**V2 建库，本文用库。** 本文调用 V2 的八要素（召回看 Context、排序看 Confidence、冲突看 Boundary）、L0–L5（只应用 L2+，按层分级召回）、质量框架（Strong 主导、Weak 仅提示）。**没有 V2 的结构化，本文无从召回；没有本文，V2 是死库存。**

### 10.2 对 MGOS / MVS / AIGO 的影响
