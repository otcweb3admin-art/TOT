## 序言：验证是 TOT 的"命门"，也是 Phase 1 的起点

审计（project-audit-v1）已指出：**按流量分成的归因是 TOT 的商业命门**——如果分不清"哪些增长是代运营带来的"，分成必生纠纷、经验必被污染、决策必失据。

Phase 0 建立了思想（知道什么 / 如何增长 / 凭什么说 / 为何越来越强）；**Phase 1 的第一件事，就是把"证明有效"这件事做成一个系统**。因为：

> **一个无法证明自己有效的增长系统，本质上还是一台营销建议机。**

MVS（Merchant Validation System）就是回答"TOT 凭什么说自己的方案真有效"的那个系统。它是分成的依据、经验的入口质检、策略的校准器。

## 1. Validation System Definition（验证系统定义）

### 1.1 什么是 MVS

> **MVS = 以"真实数据"为唯一依据，按"证据标准（而非绝对确定性）"证明增长结果真实、且可归因于 TOT 方案的验证子系统。**

它把"我们以为有效"转化为"有据证明有效"，并据此驱动三件事：**分成结算、经验准入、策略校准**。

### 1.2 MVS 验证四件事（层层递进）

```
① 执行验证：方案真的被执行了吗？（没执行 → 谈不上归因）
        ▼
② 结果验证：增长真的发生了吗？（真实数据 E1，非方案数据）
        ▼
③ 归因验证：增长真的来自我们吗？（vs 自然/季节/节日/商家自身）★命门★
        ▼
④ 价值验证：这增长值吗？（KPI 达成 + ROI 真实）
```

**只有四道全过，一个增长结果才"成立"**——才能用于分成、才配升级为经验、才作为策略依据。

### 1.3 为什么是 Phase 1 起点

- **它是命门**：分成靠它，不先做它，商业闭环跑不通。
- **它是经验的入口**：增长智能（Phase 0 第六份）的经验库，质量取决于 MVS 的把关；没有 MVS，经验库会被"假成功"污染。
- **它把 Phase 0 落到实处**：Evidence Framework 的 E1/验证、MGOS 的 Validation 段，都在 MVS 里变成可运行的机制。

## 2. Validation Philosophy（验证哲学）

MVS 的可信，来自一套**反直觉但诚实**的哲学：

| 原则 | 含义 | 为什么 |
|------|------|--------|
| **默认无效（举证责任在"声称"方）** | 在证据出现前，**默认增长不是 TOT 带来的** | 防止"贪天之功"；像科学的零假设、法律的举证责任 |
| **诚实优先于完整** | **宁可少算（漏掉部分真实贡献），不可多报（虚报）** | 少算只损失一点分成，多报会摧毁信任与经验库 |
| **真实数据唯一** | 只用 E1 真实数据，**禁用方案数据代替结果** | 呼应 TB-007 铁律 |
| **可追溯优于估算** | 优先用"硬可追踪"的归因，其次才用统计估算 | 可追踪不扯皮，估算有争议 |
| **承认不确定性** | 标注置信度，**绝不把估算当确定** | 过度自信是归因谬误的温床 |
| **主动反归因谬误** | 每次归因前，**主动寻找替代解释**（季节/节日/自然/商家自身）| 不是"找证据支持自己"，而是"先排除不是自己" |

> **核心姿态**：MVS 不追求"把所有增长都算成自己的"，而追求"只认领能证明的那部分"。**这种保守和诚实，本身就是产品**——它让分成口径清晰、经验库干净、商家信任稳固（呼应 SYSTEM_BLUEPRINT §8："口径清晰、不扯皮，让合作先稳稳跑起来"）。

## 3. Baseline Framework（基线框架）

### 3.1 为什么需要基线

要衡量"增量"，必须有一个"增长前"的参照系。**没有基线，就分不清"涨了"还是"本来就这样"。**

### 3.2 基线在 TB-001 建立

TB-001 的 O-6"增长前基准数据"（日均客流、月营业额、粉丝、咨询量）**就是基线锚点**——MVS 在商家接入时即固化这组数据。

| 基线类型 | 内容 |
|---------|------|
| 客流基线 | 代运营前日均/月客流 |
| 营业额基线 | 代运营前月营业额 |
| 线上基线 | 代运营前粉丝/流量/咨询量 |
| 渠道基线 | 各渠道代运营前的贡献 |

### 3.3 基线校正（避免被假象误导）

| 校正 | 做法 | 防止 |
|------|------|------|
| **多点基线** | 取代运营前 N 个月均值，**不取单点** | 单点偶然波动误导 |
| **同比锚定** | 与**去年同期**比，而非上月比 | 季节性混入增量 |
| **入场时点声明** | 标注旺季/淡季入场 | 旺季入场高估、淡季入场低估 |

> **基线是归因与 ROI 的共同锚**。基线不准，后面全错——所以基线必须真实（E1）、多点、同比校正。

## 4. Execution Verification Framework（执行验证框架）

### 4.1 先验证执行，再谈归因

> **不能把结果归因于一个根本没被执行的方案。** 如果内容没发、活动没办，那"增长"就不是"内容方案/活动方案"带来的。

执行验证回答四问：**做了吗 / 做到什么程度（vs 计划）/ 何时做 / 花了多少**。这正是知识架构里的"执行知识（E1）"。

### 4.2 为什么执行验证是归因的前置闸门

- **区分"方案没效"与"方案没执行"**：很多"失败"其实是没执行到位，而非策略错（呼应 TB-007"执行问题 vs 策略问题"）。
- **没有执行记录，结果无法归因**：归因分析只接纳"已执行且达标"的方案动作。
- **执行偏差要量化**：计划发 30 条、实际发 8 条，则该方案的效果判断必须按"实际 8 条"而非"计划 30 条"。

> 执行验证是归因的"准入门"：**未验证执行的动作，不进入归因分析。**

## 5. Growth Attribution Framework（增长归因框架）★全文核心★

### 5.1 直面难题：单商家无对照组

一家实体店**没有平行宇宙、没有对照组**，无法做严格的随机对照实验（RCT）。因此：

> **MVS 不声称"100% 证明因果"，而是用"分层归因 + 混淆排除 + 置信标注"，把归因做到"证据标准下可信、且不扯皮"。** 这是诚实，也是唯一可落地的路。

### 5.2 三层归因法（置信度递减、覆盖度递增）

#### 第一层：可追踪来源法（主口径，2026-06-01 决策）★高置信·保守·不扯皮★
**只认领有"数字足迹"可追溯到 TOT 线上工作的增长**：

| 可追踪来源 | 为什么可归因 |
|-----------|-------------|
| 专属优惠码核销 | 该码只在 TOT 建的账号/内容发放 → 用码者必来自线上 |
| 线上预约到店 | 预约动作发生在 TOT 建的渠道 |
| 私信咨询转成交 | 咨询入口是 TOT 建的账号 |
| Google Maps 导航点击/搜索到店 | TOT 做的本地 SEO 直接驱动 |
| 专属落地页/二维码/团购链接 | 唯一来源是线上投放 |

**逻辑**：用码的顾客、点专属链接的顾客，**有明确的因果路径**，绕开了反事实难题。**保守（会少算自然溢出的口碑客），但每一笔都站得住、不扯皮**——这是分成起步的主口径。

#### 第二层：基线增量法（参考口径）★高覆盖·高争议★
**增量 = 实际结果 − 基线（同比校正后）**，并**显式声明混淆变量**。

- 用途：**用于讨论与趋势参考，起步不作为硬分成基数**（呼应蓝图：先用可追踪来源法，信任建立后再评估引入基线增量法）。
- 必须同比（控季节）+ 多点基线 + 列出未控制的混淆。

#### 第三层：反事实推理（逻辑层）
回答"没有我们，这增长会发生吗？"，靠三类信号：
- **时间相关性**：增长与 TOT 动作同期（某条内容爆 → 当周咨询激增）。
- **渠道特异性**：TOT 建的线上渠道在涨，而无线上足迹的自然客流持平。
- **机制可信性**：存在可解释的路径（看到内容 → 搜索 → 到店）。

### 5.3 混淆变量排除（直接回答任务的核心问题）

> **每一次归因前，主动排除四类替代解释：**

| 替代解释 | 是什么 | MVS 如何控制 |
|---------|--------|-------------|
| **自然增长** | 商家本身的增长趋势 | 多点基线 + 同比；可追踪来源法天然排除（只认线上足迹）|
| **季节增长** | 行业季节性（餐饮旺季）| **同比**（去年同期）+ 季节校正 |
| **节日增长** | 节假日带来的临时客流 | 节日校正 + 与往年同节日比 |
| **商家自身能力增长** | 商家改了产品/服务/自己投了广告/开了新店 | **执行变量声明**：记录商家自身的非 TOT 动作，隔离扣除 |

> **关键**：可追踪来源法之所以是主口径，正因为它**结构上免疫这四类混淆**——一个用了"只在 TOT 抖音号发放的优惠码"的顾客，不可能是自然/季节/节日/商家自身带来的。这就是它"不扯皮"的根本原因。

### 5.4 规模带来归因能力（跨商家准对照）

单商家无对照，但**当商家足够多时，相似商家形成"准对照组"**：做了某 play 的一批 vs 没做的一批，结果差异提供了**准实验证据**。这是增长智能（1000 家 > 10 家）在归因上的体现——**规模越大，归因越强**。

### 5.5 归因结论的诚实结构

```
归因结论 = 可追踪部分（高置信，认领）
        + 估算增量部分（标置信 + 混淆声明，供参考）
        + 不可归因部分（诚实标注"无法证明来自我们"）
```

> **绝不把"不可归因"偷偷算进"我们的功劳"。** 这条诚实，是 MVS 区别于"营销吹嘘"的分水岭。

## 6. KPI Validation Framework（KPI 验证框架）

- **验证对象**：TB-008 设定的 KPI（基于 TB-001 成功标准 + 真实基线、**按行业切片、非通用默认值**）对照真实结果。
- **验证内容**：完成度（实际/目标）、数据充分性、真实性（E1）。
- **行业差异**：牙科 KPI（咨询/线索/到店）≠ 餐饮 KPI（到店/复购/翻台）——**KPI 标准随行业切片**（呼应增长方法论行业分类）。
- **防止**：KPI 造假（方案数据冒充）、数据不足却下确定结论、套通用默认值。

> KPI 验证回答"达没达到当初定的目标"，且这个目标本身必须是**真实、可量化、对齐漏斗环节、按行业校准**的——否则"达标"也无意义。

## 7. ROI Validation Framework（ROI 验证框架）

> 投流（P-001）是**最干净的归因样本**——有明确花费、有可追踪转化。

| 验证项 | 做法 |
|-------|------|
| **真实获客成本** | 实际投流花费 / 实际带来的可追踪客户，对照 TB-001"可接受最高获客成本" |
| **真实 ROI** | 实际回报 / 实际花费，对照 TB-001 ROI 预期 |
| **测试期验证（P-001 四步）** | 小额测试先验证获客成本 ≤ 可接受，再放量——**验证内建在投流结构里** |
| **隔离自然客** | 投流客户用专属码/落地页隔离，**扣除"本来就会来的"** |

- **ROI 低于预期** → 触发 TB-007 复盘调整（降预算/换平台/改素材/暂停）。
- **ROI 验证喂养投流经验**（切片：平台 × 预算档 × 行业），是增长智能投流经验的来源。

> ROI 验证是"价值验证"的核心：增长不仅要发生、要可归因，还要**值**——花 $1 投流带来的回报，必须对得起预期。

## 8. Experience Admission Framework（经验准入框架）

> **MVS 是经验库的入口质检员。** 增长智能（Phase 0 第六份）的经验库质量，取决于 MVS 在这里把多严的关。

### 8.1 只有"通过验证"的结果才配成为经验

```
执行已验证 + 结果真实(E1) + 归因高置信 + 多案例同场景一致可复现 + 人工审定
        ▼
才允许升级为经验（E3 / L2+）
```

### 8.2 验证不足者，禁止入库

- ❌ **未验证的成功**：不得升级经验（呼应证据宪法 §8："一个案例成功 ≠ 经验"）。
- ❌ **低置信归因的"成功"**：把"季节增长"误当"我们的成功模式" = **对智能库的致命污染**。**归因置信不达标，结果不入经验库。**
- ✅ **失败同等准入**：被验证为"策略问题"的失败 = 高价值失败模式经验。

> **MVS 与 Growth Intelligence 的接口就在这里**：MVS 负责"验证 + 高置信归因"，达标的才流入经验库。**验证是经验的守门人，归因置信是那把锁。**

## 9. Failure Analysis Framework（失败分析框架）

> **失败被诚实记录与分析，而非掩盖。** 这是 TOT 可信的标志，也是防止智能库幸存者偏差的关键。

### 9.1 失败三归因

| 失败类型 | 含义 | 下一步 |
|---------|------|--------|
| **执行问题** | 方案没被执行到位 | 调整执行，**不否定策略** |
| **策略问题** | 执行到位了仍无效 | 改策略 / 重做，**沉淀为失败模式经验** |
| **外部因素** | 季节/市场/商家自身导致 | 记录但**不否定方案**（非方案之过）|

> 区分这三者决定下一步——**错把"执行问题"判成"策略重做"，会丢掉一个本来有效的方案**（呼应 TB-007 防误判重做）。

### 9.2 失败是高价值资产

- **失败模式与成功模式同等沉淀**（"在 X 条件下做 Y 会失败"是宝贵 L2/L3 经验）。
- **主动记录失败 = 防幸存者偏差**：若只记成功，智能库会系统性高估打法有效性（呼应增长智能治理）。
- **诚实失败分析是信任的基石**：敢对商家说"这个没效，原因是 X，下一步这样调"，比掩盖更能建立长期信任。

## 10. Architecture Conclusions（架构结论）

1. **MVS 是 TOT 的证明引擎**：以真实数据、按证据标准证明增长真实且可归因，驱动分成、经验准入、策略校准。
2. **验证四件事层层递进**：执行 → 结果 → 归因 → 价值，四道全过结果才"成立"。
3. **验证哲学是诚实**：默认无效、诚实优先于完整、宁少算不多报、承认不确定、主动反归因谬误。
4. **基线是归因与 ROI 的共同锚**：在 TB-001 建立，须真实、多点、同比校正。
5. **执行验证是归因的前置闸门**：没执行就没归因；要区分"没效"与"没执行"。
6. **归因用三层法**：可追踪来源法（主口径·高置信·不扯皮）+ 基线增量法（参考·高争议）+ 反事实推理；**主动排除自然/季节/节日/商家自身四类混淆**。
7. **可追踪来源法结构上免疫四类混淆**——这是它"不扯皮"、成为分成主口径的根本原因。
8. **规模带来归因能力**：足够多的相似商家形成准对照组，1000 家 > 10 家在归因上同样成立。
9. **归因结论诚实分三块**：可追踪（认领）+ 估算（标置信）+ 不可归因（诚实标注），绝不偷算功劳。
10. **MVS 是经验库的守门人**：只有"高置信归因 + 多案例一致可复现"的结果才入库；低置信"成功"会污染智能。
11. **失败被诚实分析**：三归因（执行/策略/外部），失败模式同等沉淀，防幸存者偏差。
12. **TOT 是增长验证系统，不仅是增长建议系统**（见下）。

### 10.1 为什么 TOT 是增长验证系统，而不仅仅是增长建议系统

| | 增长建议系统 | 增长验证系统（MVS）|
|---|------------|-------------------|
| 输出 | 给方案 | 给方案 **+ 证明方案有效** |
| 对结果 | 不证明 | **按证据标准证明真实 + 可归因 + 有 ROI** |
| 面对"增长" | 默认归功于自己 | **默认不是自己，举证后才认领** |
| 面对季节/节日/自然/商家自身 | 含糊带过 | **主动排除，诚实标注不可归因部分** |
| 分成 | 易扯皮 | **可追踪来源法，口径清晰** |
| 经验 | 易被假成功污染 | **验证 + 高置信归因 才入库** |

> **最终回答**：增长建议系统只说"该做什么"，做完之后增长是不是自己的功劳、是真有效还是季节/节日/自然/商家自身的假象，它不回答。**MVS 回答这个问题**——它用基线、执行验证、可追踪归因、混淆排除、ROI 校验，**把"增长发生了"诚实地拆成"哪些是我们的、哪些不是、有多可信"**。正因为 TOT 敢于、且有系统地证明（并诚实承认不可归因的部分），它才不只是增长建议系统，而是 **增长验证系统**。
