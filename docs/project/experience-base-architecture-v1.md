## 序言：经验库是"机器"，增长智能是它运转的"产物"

TASK-005 回答了"经验为什么是护城河"，TASK-007 回答了"什么样的结果配进经验库"。本文回答**中间那台机器怎么运转**：经验如何在库里被组织、升级、复用、淘汰。

> **Experience Base（经验库）是体系与运作机制；Growth Intelligence（增长智能）是这台机器运转良好后涌现的能力。**
> 库管不好，再多经验也只是一堆资料；库运转得好，经验才升华为能为任意新商家精准供能的智能。

这台机器决定 TOT 的命运分层：停在"攒案例"，还是升到"建经验"，最终成为"有智能"。

## 1. Experience Base Definition（经验库定义）

### 1.1 什么是 Experience Base

> **Experience Base = TOT 沉淀、组织、治理、复用"被验证有效（及无效）的增长经验"的资产运作体系。** 它是 Growth Intelligence 的载体与运作机制。

- **输入**：MVS 验证通过、**高置信归因**的去标识结果（接 TASK-007 出口）。
- **输出**：可召回、可应用、带置信的增长经验 → 为新商家供能。
- **管理对象**：Experience Asset（资产目录第六类）的全生命周期。

### 1.2 经验库的四大职能

```
沉淀(Accumulate) ──► 升级(Promote) ──► 复用(Reuse) ──► 淘汰(Retire)
  把验证结果入库     案例结晶为经验    为新商家召回    过期经验降权退役
       ▲                                  │
       └────────── 复用产生新案例，回流强化 ◄┘
```

### 1.3 Experience Base ≠ Growth Intelligence

| | Experience Base | Growth Intelligence |
|---|----------------|---------------------|
| 是什么 | 经验的库 + 运作机制 | 库运转好后涌现的能力 |
| 形态 | 资产体系 | 可召回、成网络、可应用的智能 |
| 类比 | 图书馆 + 管理制度 | "会用全馆藏书的专家" |

> **建好库 ≠ 有智能**——这正是本文与"1000 条经验 ≠ 增长智能"要回答的核心。

## 2. Experience Lifecycle（经验生命周期）

每条经验资产在库中的完整旅程（**是活资产，持续被强化或弱化，非一次性终结**）：

```
候选 ──► 案例(L1) ──► 去重合并 ──► 经验(L2+) ──► 复用中(Active) ──► 降权 ──► 退役
Candidate   Case        Dedup       Experience      Reuse         Deprecate  Retire
(MVS标记)  (去标识入库)  (同模式归并)  (升级结晶)     (召回应用)    (时效/被质疑) (归档保留)
   │                                     ▲   │                                  │
   │          新案例确认 → 强化置信 ──────┘   │ 复用产生新案例(回流)               │ 永不物理删除
   └─────────────────────────────────────────┘                                  ▼
                                                                         保留历史版本(可追溯)
```

| 状态 | 含义 | 转移条件 |
|------|------|---------|
| **候选 Candidate** | MVS 验证通过、待入库 | 高置信归因 + 去标识 |
| **案例 Case (L1)** | 去标识入库的单次观察 | 真实结果(E1) + 打全切片标签 |
| **经验 Experience (L2+)** | 多案例结晶的模式 | 升级门槛 + 人工审定 |
| **复用中 Active** | 被召回应用、持续强化/弱化 | 上下文匹配被调用 |
| **降权 Deprecate** | 置信下降，重验前不作强依据 | 时效到期/被质疑 |
| **退役 Retire** | 长期失效/被推翻，移出活跃召回 | 重验不过/被新证据推翻 |

> **关键**：经验不是"沉淀完就静止"——新案例会**强化**它（置信升），矛盾案例或时间会**弱化**它（置信降）。它是**活资产**。

## 3. Case → Experience → Intelligence Model（核心转化模型）

这是回答"100 案例 ≠ 100 经验"与"1000 经验 ≠ 增长智能"的中心。

```
案例 Case ──(过滤+去重+升级)──► 经验 Experience ──(结构+召回+网络+治理)──► 增长智能 Intelligence
  多、原始        数量不守恒          少、被验证             质变非堆叠            能力、护城河
```

### 3.1 为什么 100 个案例 ≠ 100 条经验（数量不守恒）

案例到经验**不是 1:1**，而是经过三道"减法"：

| 减法 | 机制 | 效果 |
|------|------|------|
| **① 质检过滤** | 未验证 / 低置信归因的案例不成经验（MVS 门槛）| 数量减少 |
| **② 去重合并** | 同场景同模式的多个案例**合并为 1 条经验**（并**增强其置信**，不是新增）| 数量减少、质量增强 |
| **③ 升级门槛** | 单案例只是 L1；要成 L2 经验需"多案例 + 同场景 + 一致 + 可复现"| 不达标者停在案例层 |

**worked example（餐饮）**：
```
100 个餐饮案例
  → 质检过滤：30 个归因低置信/未验证 → 剔除，剩 70
  → 去重合并：70 个收敛为 8 种不同模式（多数是同模式的重复印证）
  → 升级门槛：仅 5 种达到"多案例一致可复现" → 升为 L2 经验
∴ 100 案例 → 5 条经验（+ 若干仍在 L1 的案例 + 被剔除的）
```

> **结论**：**案例是原料，经验是被验证、去重、升级后的成品。** 100 案例往往只结晶出个位数的强经验——而且这是**好事**：重复印证让那几条经验更硬。**数量减少 = 质量压缩，不是浪费。**

### 3.2 为什么 1000 条经验 ≠ 增长智能（质变非堆叠）

1000 条**散落、无结构、无召回、互相矛盾、混着过期**的经验，只是"一堆资料"。增长智能要求经验之上再叠五层：

| 缺了它就不是智能 | 含义 |
|-----------------|------|
| **① 结构化切片** | 经验按上下文（行业×城市×客单×渠道×目标）建立多维索引，**可被检索** |
| **② 召回机制** | 能根据新商家的上下文，**匹配出**相关经验 |
| **③ 经验网络** | 经验之间有关系：抽象（L3/L4/L5）、冲突仲裁、组合 |
| **④ 自我治理** | 质量加权、过期降权、矛盾仲裁，**库保持干净** |
| **⑤ 可应用性** | 能带着置信去**驱动新商家的诊断与策略** |

> **类比**：1000 本书 ≠ 一个会用它们的专家。**经验是书，智能是"知道何时翻哪本、并综合判断"的能力。** 增长智能 = 经验 + 结构 + 召回 + 网络 + 治理 + 可应用——是**质变**，不是把经验堆到 1000 条。

## 4. Experience Classification Framework（经验分类框架）

经验必须**多维索引**，否则无法被召回（这正是"经验→智能"的结构前提）。每条经验沿多个轴同时分类：

| 分类轴 | 取值 | 作用 |
|--------|------|------|
| **层级** | L1 案例 / L2 经验 / L3 行业 / L4 跨行业 / L5 规律 | 抽象度与适用范围（接 TASK-005）|
| **切片维度** | 行业 × 城市/国家 × 客单价 × 渠道 × 内容类型 × 引流方式 × 投流 × 增长目标 | 上下文索引，决定能否被正确召回 |
| **极性** | 成功模式 / 失败模式 | **同等价值**（失败模式防踩坑）|
| **能力/环节** | 获客/信任/转化/复购/品牌 · 漏斗阶段 · 杠杆 | 对齐增长方法论，便于按需调取 |
| **置信** | 强 / 中 / 弱 | 决定复用权限（重决策须强经验）|

> **多维索引 = 可召回性的前提**。一条没有切片标签的经验，**要么召回不到（死库存），要么被误召回（污染）**。分类不是整理癖，是智能的地基。

## 5. Experience Quality Framework（经验质量框架）

> **质量决定一条经验"配不配被信任、能不能驱动重决策"。** 深化 TASK-005 的质量模型为**可运营的质量态**。

### 5.1 质量五维（动态，非静态）

| 维度 | 含义 | 强 | 弱 |
|------|------|----|----|
| 样本量 | 多少独立案例支撑 | 足够多 | 单例/极少 |
| 一致性 | 多案例结果是否同向 | 高度一致 | 相互矛盾 |
| 时效性 | 是否仍成立 | 新近 | 陈旧 |
| 可复现性 | 同条件能否重复 | 可复现 | 偶然 |
| **多样性/覆盖度** | 来源是否独立多样 | 多商家多场景 | 单商家垄断 |

### 5.2 质量是动态的

经验质量**随时间与新案例变化**：新案例印证 → 质量升；矛盾案例或时间流逝 → 质量降。**质量不是入库时一锤定音，而是持续重估。**

### 5.3 质量分级决定复用权限

| 质量级 | 可做什么 |
|--------|---------|
| **强经验** | 可独立支撑**重决策**（投流/放量/改定位的依据）|
| **中经验** | 须佐证后用；可作主要参考 |
| **弱经验** | 仅作提示/候选；**不得单独支撑重决策**，须人工判断 |

> **质量 ≠ 数量**：100 个同源案例（一个商家）< 5 个多样独立案例。**经验库追求"质量加权的经验"，不是案例计数。**

## 6. Experience Promotion Framework（经验升级框架）

经验在 L1→L5 阶梯上**靠证据累积赚取升级**，**禁止自动升级、禁止越级**。

| 升级 | 门槛（须人工审定）| 防止 |
|------|------------------|------|
| **L1 → L2** | 多案例 + 同场景 + 结果一致 + 可复现 | 单例当规律 |
| **L2 → L3** | 行业内多个场景切片均验证一致（行业内抽象）| 局部当行业 |
| **L3 → L4** | 多个**独立行业**各自验证一致（跨行业）| 行业当普适 |
| **L4 → L5** | 跨行业 × 城市 × 时间反复验证、极少被证伪 → **回写方法论铁律** | 把局部经验误立为律 |

### 6.1 升级三原则

1. **证据驱动，非时间/人意驱动**：升级靠"被验证的次数与多样性"，不靠"放久了"或"谁说的"。
2. **人工审定**：AI 只能标"升级候选"，人裁定（接证据宪法 + AI 不拍板）。
3. **可降级（Demotion）**：高层经验被新证据证伪 → **降级或标注边界**（如 L4 被某新行业证伪 → 降回 L3 或细化适用条件）。

### 6.2 防越级污染（最危险的错误）

> 把一条 **L2 餐饮经验**直接当 **L4 跨行业经验**用，是经验库最致命的污染——它会让系统"自信地把餐饮打法套到牙科上"。**升级必须逐级、带验证、守切片边界。**

## 7. Experience Retirement Framework（经验淘汰框架）

> **不淘汰过期经验 = 用昨天的地图走今天的路。** 淘汰让经验库"保鲜"，是库健康的关键。

### 7.1 退役触发

| 触发 | 例子 |
|------|------|
| 时效到期 | 平台推荐/广告规则变 → 相关渠道/投流经验失效 |
| 环境变化 | 市场/行业周期变 → 旧趋势经验失准 |
| 被新证据推翻 | 新案例系统性与经验矛盾 |
| 长期未被验证 | 久未有新案例印证 → 置信自然衰减 |

### 7.2 三档处置：降权 / 退役 / 永不删除

| 处置 | 含义 |
|------|------|
| **降权 Deprecate** | 置信下调，重验前不作强依据，仍可参考 |
| **退役 Retire** | 移出活跃召回，但**归档保留** |
| **永不物理删除** | 保留历史版本，用于解释"当时为何这样判断"（接资产保留策略 + 证据失效政策）|

### 7.3 重新验证机制

到期/被质疑的经验 → **触发重验**：用最新真实结果校验 → 通过则恢复/更新置信，不过则退役。**每一轮新交付（TB-007）都顺带反向校验所引用的经验是否仍成立。**

## 8. Experience Reuse Framework（经验复用框架）

> 复用是经验库的**价值实现端**，也是飞轮的"消费"侧 + 新商家**冷启动**的来源。

### 8.1 复用机制（业务层，非算法）

```
新商家上下文（行业×城市×客单×渠道×目标）
        │ 匹配
        ▼
召回同场景经验（精确匹配优先）
        │
        ├─ 有精确匹配 → 调用对应强经验，指导诊断/策略
        ├─ 无精确匹配 → 降级到上层（L3 行业经验）作参考
        └─ 完全无匹配 → 诚实标"无经验，靠方法论起步"（不硬套）
```

### 8.2 复用四原则

1. **上下文必须匹配**：餐饮经验不喂牙科；**越精确匹配，置信越高**。
2. **复用强度按质量**：强经验可指导重决策；弱经验仅作提示，须人工判断。
3. **复用须经人工**：AI 召回 + 建议，人决定是否采用（不 AI 拍板）。
4. **无匹配时诚实**：宁可说"这个场景我们还没经验，先靠方法论"，**不可硬套不相关经验**（防误迁移）。

### 8.3 复用回流（闭环）

复用 → 指导新商家执行 → 产生新结果 → 经 MVS 验证 → 新案例回流 → **强化或修正原经验**。**用得越多，经验越准**——这是飞轮在经验库层的具体转动。

> **复用是"1000 家 > 10 家"的兑现点**：新商家因复用前人经验而**开局即有据**，成功率系统性提升。

## 9. Experience Governance Framework（经验治理框架）

> **治理是经验库的免疫系统。规模越大，污染破坏力越大。** 深化 TASK-005 的治理为可运营机制。

| 治理域 | 机制 | 防止 |
|--------|------|------|
| **准入治理** | 只收 MVS 高置信归因 + 验证通过的结果 | 假成功入库 |
| **升级治理** | 每次 promotion 人工审定、逐级、守切片 | 越级 / 过度泛化 |
| **质量治理** | 多样性门槛（防单源垄断）+ 失败同沉淀（防幸存者偏差）| 库被单一视角/只学成功带偏 |
| **时效治理** | 到期重验 + 降权退役 | 用过期经验指导当下 |
| **冲突治理** | 两经验矛盾 → 按置信/时效/样本仲裁，或**判定"切片不够细"→ 细分场景** | 矛盾经验并存误导 |
| **追溯治理** | 每条经验可追溯到支撑它的去标识案例（E1 基底）| 不可核查的"经验" |

### 9.1 冲突治理的关键洞察

> 当两条经验矛盾（"做 X 有效" vs "做 X 无效"），**多数时候不是经验错，而是切片不够细**——它们其实属于不同场景。**冲突是细分场景的信号**，治理应优先"加切片维度"而非"二选一删除"。

### 9.2 治理角色

- **AI**：标候选、做审计（查证据/查切片/查矛盾），**不创造经验、不自行升级**。
- **人**：审定升级、裁决冲突、确认退役。

> 没有治理的经验库，**规模越大越是"自信地错误"**。治理是让经验库可信地变大的唯一办法。

## 10. Architecture Conclusions（架构结论）

1. **Experience Base 是机器，Growth Intelligence 是产物**：建好库不等于有智能。
2. **经验是活资产**：被新案例强化、被时间/矛盾弱化，持续重估，非一次性沉淀。
3. **100 案例 ≠ 100 经验**：经过质检过滤 + 去重合并 + 升级门槛三道减法，100 案例常只结晶个位数强经验——数量减少是**质量压缩**，且重复印证让经验更硬。
4. **1000 经验 ≠ 增长智能**：智能 = 经验 + 结构化切片 + 召回 + 网络 + 治理 + 可应用，是**质变非堆叠**（书 ≠ 会用书的专家）。
5. **多维分类是召回的前提**：无切片标签的经验是死库存或污染源。
6. **质量是动态的、分级的**：强经验才可驱动重决策；质量 ≠ 数量（同源 100 < 多样 5）。
7. **升级靠证据逐级赚取**：禁止自动升级、禁止越级；可降级；越级泛化是最致命污染。
8. **淘汰让库保鲜**：到期重验、降权/退役、永不物理删除（保留可追溯历史）。
9. **复用是价值实现 + 冷启动**：上下文匹配、按质量定强度、人工采用、无匹配时诚实；复用回流强化经验。
10. **治理是免疫系统**：准入/升级/质量/时效/冲突/追溯六域；冲突常是"切片不够细"的信号。
11. **AI 标候选 + 审计，人审定 + 裁决**：经验库同样不 AI 拍板。

### 10.1 最终回答：TOT 如何从案例公司 → 经验公司 → 增长智能公司

| 阶段 | 特征 | 价值在 | 多数同行止步于此？ |
|------|------|--------|------------------|
| **案例公司** | 攒了很多交付案例，但**孤立、靠人记忆/口口相传、不可复用** | **人**（牛人离职即流失）| ✅ 绝大多数代运营 |
| **经验公司** | 把案例**验证、去重、升级**成结构化经验库，**可复用、可传承、不依赖单人** | **经验库** | 少数 |
| **增长智能公司** | 经验被**切片、召回、成网络、自治理**，能为**任意新商家精准匹配打法、迁移、预测** | **增长智能（护城河）** | 极少 |

**两次跃迁，各由本文的机器驱动**：
- **案例公司 → 经验公司**：靠 **MVS 验证（TASK-007）+ 本文的升级/质量框架**——把"散案例"炼成"硬经验"。
- **经验公司 → 增长智能公司**：靠**本文的分类/召回/复用/治理框架**——把"经验堆"组织成"可应用的智能网"。

> **最终判断**：TOT 不甘于做"案例公司"（价值锁在人脑里、随人流失）。它要用 **验证 + 经验库运作机制**，先成为"经验公司"（价值沉淀在库里、可传承），最终成为"**增长智能公司**"（价值在于能为任意新商家供能的智能网络——这才是不可复制、越用越强的护城河）。**而这台机器，就是 Experience Base。**
