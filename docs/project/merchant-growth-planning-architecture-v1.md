## 序言：计划是策略的落地路径

> **计划不是愿望清单，不是任务堆叠，不是把策略写得更细。计划是把已确认的增长策略，转化为可执行、可监督、可验证、可复盘的行动系统。**

- **策略决定方向**（优先打哪一仗）。
- **计划决定路径**（这仗具体怎么打）。

如果计划错误：执行会失控 → 责任会模糊 → 进度会失真 → 验证会失败 → 经验无法沉淀。因此**增长计划是 MGOS 从策略进入执行的第三道质量门**。

## 第一章　Growth Plan Definition（增长计划定义）

### 1.1 计划不是什么

> 不是**愿望清单**（想做的事）、不是**任务堆叠**（一堆无结构的动作）、不是**把策略写细**（策略+细节 ≠ 计划）。

### 1.2 计划是什么

> **计划 = 把已确认策略，转化为"可执行、可监督、可验证、可复盘"的行动系统。**

四个特征：

| 特征 | 含义 |
|------|------|
| **可执行** | 能落地（有责任、有资源、有节奏）|
| **可监督** | 有进度/责任，能查偏离 |
| **可验证** | 有验证节点 + 数据采集 |
| **可复盘** | 留下可归因的数据 |

### 1.3 五层区分（关键）

| 层 | 回答 |
|----|------|
| **策略 Strategy** | 优先方向（打哪一仗）|
| **计划 Plan** | 行动路径（怎么打——阶段/节奏/资源/验证节点）|
| **任务 Task** | 具体动作（做什么——单个可执行单元）|
| **执行 Execution** | 真实完成情况（做了没/做到什么程度）|
| **验证 Validation** | 是否有效 |

> **计划是"系统"不是"清单"**：清单是平的动作集合；系统有**结构**（阶段/层级/节奏）、有**控制**（质量门/检查点）、有**反馈**（验证/复盘）。

## 第二章　Strategy-to-Plan Bridge（策略到计划的桥）

> **不得脱离策略直接生成计划**——脱离策略的"计划"是任务堆叠、是瞎忙。

计划必须承接策略（TASK-017）的全部输出：

| 策略输出 | 计划如何承接 |
|---------|-------------|
| 已确认策略（经人确认）| → 计划展开的前提（未确认策略不进计划）|
| 策略目标 | → 计划各阶段目标的对标 |
| 适用条件 / 边界 | → 计划的约束条件 |
| 所需资产 / 资源 | → 计划的资产建设/资源配置动作 |
| 证据依据 | → 关键动作的依据继承 |
| 历史经验 | → 计划节奏/打法参考（须适配）|
| 风险 | → 计划的风险预案 |
| 验证方式 | → 计划的验证节点 |
| 人工确认点 | → 计划确认的前置 |

> **早期控制变量**：一个策略 → 一个计划；多策略并行需分别计划，但早期聚焦少数（承 pilot 的最小验证闭环），避免变量过多无法归因。

## 第三章　Plan Input Model（计划输入模型）

计划比策略更"现实约束敏感"——它要落到真实的人、时间、预算、产能。至少依赖十二类输入：

| # | 输入 | 作用 | 缺失后果 |
|---|------|------|---------|
| 1 | 诊断结果 | 计划要解决的根因 | 偏离真问题 |
| 2 | 已确认策略 | 方向 | 瞎忙 |
| 3 | 商家资产 | 能用什么打 | 计划做不到 |
| 4 | 执行资源 | 谁来做 | 落不了地 |
| 5 | 时间周期 | 节奏 | 无 deadline |
| 6 | 预算约束 | 花多少 | 超支/无法执行 |
| 7 | **内容产能** | 能产多少内容 | 排期超产能、断更 |
| 8 | 渠道状态 | 阵地是否就位 | 计划悬空 |
| 9 | 验证条件 | 能否验证 | 做了不知有没有用 |
| 10 | 风险边界 | 红线 | 撞红线 |
| 11 | **商家配合条件** | 商家能配合到什么程度 | 计划落不了地 |
| 12 | 人工确认事项 | 需人定的点 | 越权 |

> **缺关键输入不得输出确定性计划**。内容产能（7）与商家配合条件（11）是计划特有的现实约束，策略阶段易忽略，计划阶段必须落实。

---

## 第四章　Planning Unit Model（计划单元模型）

计划必须拆成可执行单元（Planning Unit）。每个单元至少包含：

| 字段 | 含义 | 缺失后果 |
|------|------|---------|
| 目标 | 这个单元要达成什么 | 无意义动作 |
| 动作 | 具体做什么 | 不知道做什么 |
| **负责人** | 谁做 | **没人做** |
| **时间** | 何时做完 | **永远不做** |
| 所需资产 | 用什么 | 做不到 |
| 所需输入 | 依赖什么前置 | 卡住 |
| 预期输出 | 产出什么 | 无法验收 |
| 证据依据 | 凭什么这么做 | E0 猜测 |
| 风险 | 可能出什么问题 | 无预案 |
| **验证方式** | 怎么证明有效 | **做了不知有没有用** |

> **铁律：没有负责人、时间、验证方式的内容不是计划，是愿望。** 计划 = 有结构组织（按阶段/周排布）的计划单元集合。

## 第五章　90-Day Growth Plan Structure（90 天增长计划结构）

> **90 天是战略周期**（一轮 MGOS）。阶段间有质量门：前阶段不达标不进下阶段。

| 阶段 | 目标 | 主要动作 | 关键产出 | 质量门 | 验证节点 |
|------|------|---------|---------|--------|---------|
| **Day 1–15 基础与准备** | 可被验证的起点 | 诊断确认/基线固化/归因口径/资源配置 | 基线 + 准备就绪 | 数据可采 + 归因口径定 | 准备度检查 |
| **Day 16–30 资产建设** | 最小必要资产就位 | 账号/内容/信任素材搭建 | Channel + Content 资产 | 资产质量验收 | 资产就位检查 |
| **Day 31–60 增长执行** | 策略主攻方向落地 + **埋可归因钩子** | 内容运营/引流/承接 | 真实运营动作 + 可追踪入口 | 执行到位度 + 承接能力 | 中期数据 |
| **Day 61–90 验证与优化** | 验证可归因增长 + 优化 | 复盘/MVS 归因/调整 | 可验证增长案例 + 下一轮方向 | 复盘可信 | 结果验证 |

> **归因钩子从 Day 31 埋**（专属优惠码/落地页/Maps），不是 Day 90 才想（承 pilot）。

## 第六章　30-Day Action Plan（30 天行动计划）

> **90 天是战略周期，30 天是执行周期。** 30 天把 90 天阶段目标落成本月可执行动作。

30 天计划必须包含：

```
本月目标 · 本月核心动作 · 内容计划 · 渠道动作 · 引流动作 · 数据采集 · 验证节点 · 风险预案
```

> **关键**：数据采集与验证节点必须在 30 天计划里**明确**（不是事后补）——否则月末无法复盘归因。30 天是"可控的执行颗粒度"：比 90 天具体、比周计划有方向。

---

## 第七章　Weekly Execution Plan（周执行计划）

> **周计划是执行控制单元**（最小监督颗粒度）。

每周必须明确：

```
本周目标 · 本周任务 · 负责人 · 交付物 · 检查点 · 数据记录 · 风险提示 · 是否偏离计划
```

### 7.1 关键字段："是否偏离计划"= 反馈控制

- 每周对照计划检查偏离，**早发现早纠偏**（而非月末才发现）。偏离要么纠回，要么走调整流程（Ch9）。
- **数据记录每周做**，不是月末补——保证验证数据真实、及时（承 MVS 真实数据）。

---

## 第八章　Plan Quality Gate（计划质量门）

一份合格计划**必须满足**：

```
☑ 承接策略   ☑ 有明确目标   ☑ 有执行路径   ☑ 有时间节奏   ☑ 有责任归属
☑ 有所需资产 ☑ 有证据依据   ☑ 有验证节点   ☑ 有风险预案   ☑ 有人工确认点
```

> 不满足 → **不得进入执行阶段**。这是 MGOS **第三道质量门**（诊断 1 / 策略 2 / 计划 3）。

**防"无法执行/监督/验证的计划"**：缺责任=没人做；缺时间=不会做；缺验证节点=做了不知有没有用；缺风险预案=出问题没退路。

---

## 第九章　Plan Review and Adjustment（计划复查与调整）★保护归因★

> **计划不是一次性文件。** 但**计划调整必须保护 MVS 的验证有效性**。

### 9.1 调整机制

| 项 | 规则 |
|----|------|
| **何时调整** | 周/月检查发现偏离、外部变化（平台规则/市场）、承接不足需降级、诊断更新 |
| **谁提出** | 任何角色（执行/运营/AI）可提出 |
| **AI 如何提供** | 提供调整依据（偏离分析 + 影响评估），**不自行改计划** |
| **人如何确认** | 项目/增长负责人确认（涉资源/方向的调整须人定）|

### 9.2 核心：调整三档，保护归因

> **中途随意改变量（换平台/换打法/加新动作），就无法归因"增长是哪个动作带来的"——验证失真**（承 MVS 控制变量）。

| 调整档 | 含义 | 对归因的影响 | 处置 |
|--------|------|-------------|------|
| **微调** | 不动关键变量（如优化文案）| 无 | 自由 |
| **纠偏** | 执行没到位 → 拉回原计划 | 无（恢复原变量）| 自由 |
| **改向** | 改变关键变量/策略方向 | **破坏前后对比** | 须人确认 + **重设验证基线、此后归因重新计算** |

### 9.3 原则

- **区分"纠偏"与"改向"**：纠偏是把执行拉回计划（保护变量）；改向是开启新实验（须重设基线）。
- **验证周期内尽量不改关键变量**；必须改时，明确标注"变量已变，此后归因重新计算"（切断前后对比）。
- **不能为短期效果随意改计划导致归因失真**（如月中突然加投流，之后的客流就分不清是内容还是投流带来的）。

> **"可调整"与"保护归因"是一对张力**：调整服务现实，但不能牺牲"能证明有效"这个根本。

## 第十章　Relationship With Other Systems（与其他系统的关系）

| 系统 | 关系 |
|------|------|
| Diagnostic | 计划落地诊断指出的根因 |
| Strategy | 计划**承接**策略（方向→路径，Ch2）|
| Evidence Framework | 关键动作挂证据，E0 禁入 |
| Growth Methodology | 用冷启动节奏/先稳后放排计划 |
| Merchant Asset Catalog | 计划含资产建设动作 |
| MGOS | 计划是**第三道质量门**；指导执行 |
| MVS | 计划内含验证节点+数据采集+**归因保护**，使 MVS 可归因 |
| AIGO | AI 拆解计划，人确认（不拍板）|
| Experience Base / Application | 召回"计划经验"（哪种计划结构有效）；计划+结果沉淀回流 |
| Merchant Success | 计划落地→真实增长→商家成功 |
| Commercial Operating | 资源/人员/预算决策在 commercial 层的人 |

**四条关系主线**：计划**承接**策略 → **指导**执行 → **支持**验证（保护归因）→ **产生**经验沉淀（计划经验）。

> **核心警示**：计划错 → 执行失控 → 进度失真 → 验证失败 → 经验无法沉淀。

## 第十一章　Architecture Conclusions（架构结论）

1. **计划是行动系统，不是任务清单**：有结构、责任、节奏、验证、反馈。
2. **策略定方向，计划定路径**；计划必须承接策略，不得脱离。
3. **计划比策略更现实约束敏感**：落到真实的人/时间/预算/产能/商家配合。
4. **计划单元三必需**：负责人、时间、验证方式——缺一即"愿望"不是计划。
5. **三级嵌套**：90 天战略周期 / 30 天执行周期 / 周执行控制；阶段间有质量门。
6. **数据采集与验证节点必须前置写入计划**，不是事后补。
7. **周计划"是否偏离"= 反馈控制**：早发现早纠偏。
8. **计划质量门**（十项）不达标不进执行——MGOS 第三道质量门。
9. **计划调整必须保护归因**：区分纠偏（拉回，保护变量）与改向（新实验，重设基线）；不为短期效果乱改变量。
10. **AI 拆解、人确认**：AI 提供拆解/资源/风险/验证节点，人定计划/资源/人员/预算/承诺并担责。

### 11.1 五个根本问题的回答

| 问题 | 回答 |
|------|------|
| **为什么计划不是任务清单？** | 清单是平的动作集；计划是有结构、责任、节奏、验证、反馈的行动系统。|
| **为什么计划必须承接策略？** | 脱离策略的计划 = 瞎忙；计划是策略的落地路径。|
| **为什么计划必须可验证？** | 不可验证的计划 = 做了不知有没有用 = 无法归因、无法沉淀经验。|
| **为什么计划调整必须保护归因？** | 中途乱改变量 → 增长分不清是哪个动作带来的 → 验证失真 → 经验污染。|
| **为什么没有计划质量门就会执行失控？** | 缺责任=没人做、缺时间=不会做、缺验证=无法监督、缺风险预案=出问题没退路。|

### 11.2 最终结论

> **增长计划是 MGOS 从策略进入执行的第三道质量门。** 策略决定方向，计划决定路径——它把已确认策略转化为**可执行、可监督、可验证、可复盘的行动系统**，AI 拆解、人确认，**且调整始终保护归因**。计划对，执行才不失控、验证才不失真、经验才能沉淀。
