## 审计结论概览（先看这一段）

1. **文档资产成熟度极高，代码成熟度近乎为零**。8 个模板已封板、形成闭环、P-001 全链兼容；但 `app/` 只有脚手架。项目真正的风险不在"想清楚没有"，而在"骨架落地时如何不丢掉模板的精度，又不过度工程"。
2. **存在一条"精度落差线"**：模板用 **Output 级 Workflow Gate + 15 项参数交接契约** 描述流转，而 `WORKFLOW_STATE_MODEL` 用 **节点级 7 状态** 描述流转。MVP 骨架（node + JSON）落在后者粒度上，**模板的 Output 级把关在骨架阶段会被压扁**。这是设计与落地之间最需要对齐的地方。
3. **商业闭环的"命门"（按流量分成的归因）在系统里目前是空白**。归因、流量来源记录、分成试算都还在"手工/系统外"，而这正是收入实现的关键环节——属于已知但未建的模块。
4. **整个交付链高度依赖"真实数据"与"商家配合"**：TB-007 复盘、分成结算都要求真实经营数据（客流/咨询/转化/营业额），但系统没有承载这些数据录入与归集的模块，商家端入口也与"不对外"的边界存在张力。
5. **AI 是 7/8 模板的草稿生成器**，每个模板都带大量"防漂移/降级/拦截空话"规则——这意味着 **Prompt 工程 + 人工审核兜底** 是 AI 子系统的真正工作量所在，远不止"调一次 API"。

# 项目定位

**一句话**：海外华人实体商家**线上增长交付的流程协作系统**——带角色权限、状态流转、两级审批、多商家并行的**内部工作台**，承载 8 段标准化作业（TB-001~008），由"人/外包/AI"灵活承接，按客流量分成获利。

| 维度 | 内容 |
|------|------|
| 服务对象 | 海外华人实体商家（老字号 + 新店），共同痛点"线上营销短板" |
| 服务内容 | 全链路代运营：诊断→账号→素材→内容→直播→引流→复盘→90天规划 |
| 系统本质 | 流程驱动的协作工作台（任务分派 + 审批 + 流转 + 状态管理）|
| 收费模式 | 按客流量分成，起步用**可追踪来源法**归因（2026-06-01 已决策）|
| 商业模式 | 组团队（内部 + 外包 + AI），流水线批量代运营，按效果（流量）付费 |

**明确的"不是"边界**（多份文档一致强调，属硬约束）：不是 CRM / BI / 财务结算系统；不是表单系统；不是自动发布工具；不是多租户 SaaS；**不是无人工审核的全自动 AI**。

**定位评价**：定位清晰、自洽、且被治理文档反复锚定（CODE.md 为最高规则）。风险不在定位摇摆，而在**落地时容易因"模板很全"而冲动做大**——治理文档已预设了"完整骨架优先、能跑就行、不提前平台化"的护栏。

# 用户角色

系统定义 **6 个角色**（ROLE_MODEL + SYSTEM_BLUEPRINT，含外包补充）：

| 角色 | 标识 | 权限等级 | 核心职责 | 审批权 |
|------|------|---------|---------|--------|
| 商家 | `merchant` | 外部 | 提供信息、确认交付物 | 无 |
| 前端采集员 | `collector` | 内部-执行 | 引导填写 TB-001、录入、确认完整性 | 信息完整性确认 |
| 运营人员 | `operator` | 内部-审核 | 审核方案类产物（诊断/账号/内容/直播）、分派执行 | 方案类初审 |
| 执行人员 | `executor` | 内部-执行 | 落地执行（账号/内容/直播/引流）、上报数据 | 无 |
| 外包团队 | `outsource` | 外部-执行 | 承接重执行模块（拍摄/剪辑），仅见自己任务 | 无 |
| 管理员/总控 | `admin` | 内部-管理 | 高风险审核（引流/复盘/90天）、争议裁决、全局调度、终审放行 | 终审 + 争议裁决 |
| AI Worker | `ai_worker` | 系统-生成 | 生成各模板草稿，**只能置 `ai_generated`** | **无（绝对）** |

**权限设计要点**：
- **最小可见原则**：每个角色只看到与自己相关的任务（采集员看不到引流方案，外包只看自己那段）。
- **两级审批**（SYSTEM_BLUEPRINT 5.3，核心）：① 下阶段初审（接收方"谁用谁把关"）→ ② 管理员终审。AI 与外包**无审批权**。
- **AI 权限天花板**：不能推进至 approved/completed、不能分派、不能跳过审核——这是贯穿所有文档的红线。

**审计发现（角色层）**：
- **R1｜`merchant`/`outsource` 是"外部角色"，但系统定位"不对外/不做商家自助入驻"**——存在张力。merchant 需要填 TB-001、确认 TB-008，outsource 需要看任务，但边界文档又说"不对外开放"。MVP 阶段大概率以**内部代填 + 内部代管**消解（采集员代商家录入、管理员代管外包），但这点需在落地时明确，否则角色模型与边界冲突。
- **R2｜ROLE_MODEL（6 角色，无 outsource）与 SYSTEM_BLUEPRINT（7 角色，含 outsource）不完全一致**——属轻微文档漂移，骨架阶段建议以 SYSTEM_BLUEPRINT 为准并回填 ROLE_MODEL。
- **R3｜`ai_worker` 非登录主体**：数据模型里不宜建为可登录用户；AI 操作的审计记录需要一种"系统执行者"表达方式。

# 核心业务闭环

## 交付闭环（流程链）

```
商家接入 → TB-001 诊断 → TB-002 账号 → TB-003 素材 → TB-004 内容
        → TB-005 直播 → TB-006 引流 → TB-007 复盘 → TB-008 90天
        ↺ 循环回 TB-007（新一轮增长周期）或结束交付
```

## 商业闭环（收入实现）

```
接入付费商家 → 流水线交付（8 模板）→ 产生可追踪线上来源客流
            → 按"可追踪来源法"归因 → 按月手工试算分成 → 收入
            → 数据反哺 TB-007 复盘 → TB-008 下一周期
```

**审计发现（商业闭环层）——这是全项目最薄弱的一环**：

- **B1｜归因是"命门"，但系统无承载**。SYSTEM_BLUEPRINT 第 8 节明确：分成靠"可追踪来源客流"（私信咨询、线上核销、线上预约、Maps 导航点击等），结算"手工试算、不入系统"。**这意味着收入实现的关键数据（流量来源、归因、分成）在 MVP 里完全没有对应模块**。短期可手工，但"系统帮团队协作干活"的价值主张里，**最值钱的一段（把钱算清楚）反而在系统外**。
- **B2｜闭环依赖商家持续提供真实数据**。TB-007 Input-5（真实客流/咨询/转化/营业额/投流数据）、分成归因都要商家配合统计。商家不配合 → 复盘失真 + 分成无据 → 闭环断裂。这是商业模式的结构性依赖，需在"合作前书面约定口径"（文档已点出）+ 系统提供低摩擦的数据录入入口（尚缺）。
- **B3｜P-001 投流分成贯穿但需真实 ROI 数据**。TB-006→007→008 的投流链（四步结构 / 复盘 / 三级分支）依赖真实获客成本与 ROI，否则全链判断悬空。

# 工作流分析

## 两套粒度的流转模型（关键发现）

| 模型 | 出处 | 粒度 | 把关单位 |
|------|------|------|---------|
| **节点级状态机** | WORKFLOW_STATE_MODEL | 7 状态：draft→submitted→ai_generated→reviewing→approved/rejected→completed | 一个 TB 模板 = 一个节点 |
| **Output 级闸门 + 参数契约** | 各 TB 模板内部 | 每个 Output 各有 Workflow Gate；模板末尾有 **15 项必传参数 Workflow Contract** 交接给下一模板 | 一个 Output |
| **两级审批** | SYSTEM_BLUEPRINT 5.3 | 节点间流转：下阶段初审 + 管理员终审 | 节点交接 |

**审计发现（工作流层）**：

- **W1｜精度落差**：模板把"质量门"开在 **Output 粒度**（如 TB-002 有 8 个 Output、各自 Gate；TB-006 有 10 个 Output + P-001 四步）。而骨架的 node + `formData(JSON)` 只能表达**节点级**状态。**直接后果**：MVP 阶段，模板里精心设计的"逐 Output 准入 + 15 参数完整性检查"会被压缩成"整节点 approved/rejected"。这不是错，但要**显式决策**：骨架阶段接受"压扁"，把 Output 级 Gate 降级为审核清单（checklist）记录在 formData 里，等"细化"阶段再结构化。
- **W2｜两级审批 vs 7 状态模型未完全咬合**。7 状态里 `approved→completed` 是单节点内；而"下阶段初审 + 管理员终审"是**跨节点**的另一层。落地时需要把"两级审批"映射成状态/审批记录（建议：`approved`=本节点完成待交接，下阶段初审 + 终审通过后才 `completed` 并激活下一节点）。**当前文档未给出这层的精确状态映射**，是 MVP 必须补的一块设计。
- **W3｜各模板状态起点不一致**（已在 WORKFLOW_STATE_MODEL 标注，属合理设计）：TB-002 起于 `draft→ai_generated`（无 submitted），TB-003 无 `ai_generated`（人工/外包），TB-007 起于 `submitted`（先要真实数据）。**骨架的状态机必须按模板配置允许的状态子集**，不能一刀切。
- **W4｜大量 Output 级人工确认会带来流程堵塞风险**（多个模板审计均提示）：每个 Output 都"AI 辅助 + 人工最终确认"，高并发多商家时审核压力大、标准易漂移。**缺审核 SLA、批量审核、审核决策矩阵**——属缺失的运营机制。

## 数据流转路径（端到端）

```
人工录入(TB-001 7模块) ──► AI生成诊断草稿(O-50/51) ──► 运营审核 ──► approved
   │                                                                    │
   └── 15参数 Workflow Contract ──────────────────────────────────────┘
                       ▼
   每个 TB 模板：上游 formData + 模板"AI生成要点"作 prompt → AI 草稿 → 人工审核 → 交接参数 → 下一模板
                       ▼
   TB-007：真实经营数据(人工录入) + 各模板方案(对标) → AI复盘 → 管理员审核
                       ▼
   TB-008：复盘结果 → AI 90天规划 → 管理员+商家确认 → 循环回 TB-007
```

**关键数据契约**：每个模板末尾的 **15 项必传参数**（Workflow Contract）是跨模板数据流的"接口定义"——这是把模板串起来的真正粘合剂，**数据模型必须显式承载它**（见下节）。

# 数据模型建议

> 原则：**骨架宽而浅（JSON 承载），但为三类"真正跨模板/跨阶段的结构化数据"预留正式表**——它们是系统价值所在，不能埋在 JSON 里查不动。

## 骨架阶段（P0/MVP）：5 张核心表（沿用 P0_FOUNDATION_PLAN，已合理）

`Profile` / `Merchant`(=工单) / `Node`(9 节点/工单, formData+aiDraft 为 JSON) / `Approval`(两级审批) / `Transition`(状态审计)。

**对 5 张表的审计修正建议**：

| 编号 | 建议 | 理由 |
|------|------|------|
| D1 | `Node` 增加 `templateCode` 的**允许状态子集**约束（按 W3） | 不同模板状态起点不同 |
| D2 | `Approval.level` 已含 `initial_review/admin_final` ✓，但需补**「下阶段初审由哪个 Node 的接收方发起」**的关联（initial_review 是对**上游**节点交付的验收） | 落地 W2 两级审批 |
| D3 | `ai_worker` 不建 Profile；`Transition.actorId` 可空 + `action` 标注"AI生成"，并加 `actorType`(human/ai/system) | 落地 R3 |
| D4 | `Node` 增加 `handoffParams`(JSON) 存放该节点交接给下游的"15 项必传参数"快照 | 显式承载 Workflow Contract（W 节关键契约）|

## 越过骨架需要尽早规划的"结构化数据"（细化阶段，但现在就要预留位置）

| 实体 | 承载什么 | 为什么不能只塞 JSON | 优先级 |
|------|---------|---------------------|--------|
| **TemplateDef（模板定义/注册表）** | 每个 TB 的 Output 清单、字段schema、AI生成要点、审核清单、Gate 规则 | AI prompt 拼装、审核清单渲染、Gate 校验都要读它；建议作**代码内静态配置/种子**，非业务表 | 高（P2-P3 需要）|
| **RealMetric（真实经营数据）** | TB-007 Input-5：客流/咨询量/转化率/营业额/粉丝/发布数/直播场次/投流数据 | 复盘、KPI、分成都要查询/对比/趋势，JSON 查不动 | 高（TB-007 + 分成命门）|
| **AttributionRecord（流量归因记录）** | 可追踪线上来源客流（私信/核销/预约/Maps 点击）→ 归因到代运营 | **分成结算的命门数据**，必须可查询、可月度聚合 | 高（商业闭环 B1）|
| **Asset（素材文件）** | TB-003 产出的素材文件清单 + 对象存储引用 | 素材是文件（图/视频），需对象存储（Supabase Storage），非 JSON | 中（TB-003 落地时）|
| **Assignment / Notification（分派与通知）** | 谁分派给谁、截止时间、待办提醒、审核 SLA | 流转/分派/审核时限需要驱动协作，否则退回微信/Excel | 中（W4 流程堵塞）|

**数据模型总建议**：骨架先用 5 表 + JSON 跑通端到端；但**在 schema 设计时就为 RealMetric / AttributionRecord 留出关系位**（哪怕先空表），因为它们关系到复盘可信度与收入实现，是"细化"阶段无法回避的结构化需求。

# AI系统建议

## AI 参与节点（精确盘点）

| 模板 | AI 是否生成草稿 | AI 生成什么 | 默认承接（SYSTEM_BLUEPRINT 6）|
|------|----------------|------------|------------------------------|
| TB-001 诊断 | ✅ | 诊断报告 + 增长方向（O-50/51 等）| AI 草稿 + 人工审核 |
| TB-002 账号 | ✅ | 平台/定位/命名/Bio/视觉/人设/Maps（8 Output）| AI 草稿 + 运营完善 |
| **TB-003 素材** | ❌（无 ai_generated 态）| —（采集清单由人工/外包）| 人工 / 外包 |
| TB-004 内容 | ✅ | 定位/栏目/比例/频率/风格/禁区/前30天（9 Output）| AI 草稿 + 人工/外包制作 |
| TB-005 直播 | ✅ | 可行性/平台/目标/形式/主题/频率/风险/Readiness（9 Output）| AI 规划 + 人工执行 |
| TB-006 引流 | ✅ | 引流/转化/私域/活动/Maps/投流P-001/30天（10 Output）| AI 方案 + 人工执行 |
| TB-007 复盘 | ✅（起于 submitted，先要真实数据）| 完成度/各维度效果/问题诊断/优化/策略判断（8 Output）| AI 分析 + 人工审核 |
| TB-008 90天 | ✅ | 90天目标/路线图/平台/内容/引流/风险/KPI/周期判断（9 Output）| AI 生成 + 管理员审核 |

**结论**：AI 在 **7/8 模板**生成草稿（仅 TB-003 不生成）；所有草稿状态恒为 `ai_generated`，**必经人工审核**，AI 无任何放行权。

## AI 系统的真正工作量（审计判断）

- **A1｜Prompt 工程是主体工作，不是"调一次 API"**。每个模板都有大段"AI 生成要点"（多条防漂移原则 + 降级方向 + 全局底线 + 空话拦截），且要把**上游 formData + 15 参数契约**注入 prompt。建议建立统一的 **`generateDraft(templateCode, upstreamData)` 接口 + 每模板的 prompt 模板（含系统提示=该模板AI要点）**，启用 prompt caching（模板要点固定、可缓存）。
- **A2｜"防漂移/拦截空话"难以靠提示词 100% 保证**，人工审核是兜底。建议把每个模板的"审核要点/退回条件"沉淀为**结构化审核清单**，在审核界面逐条勾选——既落地 REVIEW_POLICY，又缓解 W4 标准漂移。
- **A3｜AI 降级链要可执行**：多个模板强调"高复杂度必须给降级方案"。AI 输出建议采用**结构化 JSON**（主方案 + 降级方案 + 风险四要素），而非纯文本，便于审核与下游消费。
- **A4｜双重 AI 职能（总管/执行）的治理**（AI_WORK_RULES）：Claude 既是项目总管又是执行者，但**决策权恒在人工**。这是工作流程约束，不影响系统架构；但系统里的"AI 生成"应与"人类操作"在审计日志中可区分（D3）。
- **A5｜模型选择**：诊断/复盘/90天规划是高推理任务（建议 Opus 级），账号/素材清单类可用更轻模型——可在 generator 接口按模板配置模型，控成本。

# 技术架构建议

## 当前技术现状

- **已定**：Next.js **16.2.6**（App Router）+ React 19 + Tailwind v4 + TS5；**Supabase**（Postgres + Auth）+ **Vercel** + **Prisma** + **Claude API**（已在 MVP/P0 计划确认）。
- **已就位**：线上仓库（已推送）、Supabase/Vercel 项目（用户已建好）。
- **未动**：`app/` 仅脚手架（layout/page 为默认欢迎页），无业务代码、无 DB 接入、无认证。

## 架构建议（基于 Next.js 16 实读结论）

| 主题 | 建议 | 依据 |
|------|------|------|
| 数据变更 | 统一用 **Server Actions**（`'use server'`），鉴权在每个 action 内首行校验 | Server Action 可被直接 POST，必须内部鉴权 |
| 授权 | 服务端 `requireRole()` 统一守卫；RLS 留到"细化"（上真实商家数据前补） | 内部工具骨架够用，避免提前复杂化 |
| 缓存 | **不启用 Cache Components**，保持动态渲染（按用户实时取数）| 内部工具、强个性化数据 |
| AI 接入 | `lib/ai/generator.ts` 接口先行 + 真接 Claude + 无 Key 降级占位 + prompt caching | A1/A2 |
| 数据库 | Prisma + Supabase 池化串（6543, pgbouncer）/ 直连串（5432）迁移；单例 Client | Vercel serverless |
| 部署 | monorepo → Vercel Root Directory = `app`；`postinstall: prisma generate` | monorepo 关键设置 |
| 存储 | 素材文件用 **Supabase Storage**（TB-003 落地时）| Asset 实体 |

## 架构层风险

- **T1｜Next.js 16 为非标准/前沿版本**（AGENTS.md 强制"写码前读 `node_modules/next/dist/docs/`"）。训练数据里的 Next 已过时，需逐特性核对文档——已部分完成，需持续遵守。
- **T2｜安全债**：① 已发生 PAT 在会话中暴露（需 revoke）；② service_role key 绝不可加 `NEXT_PUBLIC_`；③ RLS 暂缺，授权全靠服务端，上线真实商家数据前必须补。
- **T3｜不提前平台化护栏**：CODE.md / SYSTEM_BOUNDARY 明令不做多租户/自动结算/对外 API——架构上**不要引入租户隔离层、计费层**，否则违反最高规则。
